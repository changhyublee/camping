# MVP 범위

## 1. 문서 목적

이 문서는 캠핑 추천 프로젝트의 v1 MVP 범위를 정의한다.

핵심 목적은 아래 두 가지다.

- 지금 당장 만들어야 하는 기능을 명확히 정리
- 지금은 만들지 않을 기능을 분명히 제외

이 문서는 아래 전제를 따른다.

- 원격 서버 없이 시작한다
- 사용자 데이터는 `./.camping-data/` 에 저장한다
- 로컬 웹 UI와 로컬 API가 분석 실행을 담당한다
- 기본적으로 로컬 `codex CLI` 를 통해 추천 결과를 생성한다
- 필요하면 OpenAI Responses API를 fallback 으로 사용할 수 있다
- GitHub는 코드/문서 저장소로만 사용한다

관련 문서:

- [`technical-architecture.md`](technical-architecture.md)
- [`local-api-contract.md`](local-api-contract.md)
- [`trip-analysis-workflow.md`](trip-analysis-workflow.md)

## 2. MVP의 정의

이 프로젝트의 MVP는 사용자가 로컬 파일에 기본 데이터를 저장해두고, 로컬 웹 UI에서 특정 캠핑 일정에 대한 trip 요청을 선택하거나 확인한 뒤, 로컬 API가 기본적으로 로컬 `codex exec` 응답을 받아 캠핑 준비물, 연령대별 개인 준비물, 요리, 이동 동선, 주변 추천까지 포함한 결과를 Markdown으로 생성하고 화면에 보여주는 최소 제품이다.

즉, MVP는 `캠핑 추천 분석이 실제로 반복 사용 가능한 로컬 웹 도구` 를 목표로 한다.

## 3. MVP에서 해결해야 하는 핵심 문제

MVP는 아래 문제를 해결해야 한다.

- 어떤 장비를 챙겨야 하는지 빠르게 판단하기 어렵다
- 계절, 날씨, 장소에 따라 준비물이 달라진다
- 성인, 유치원생, 고등학생처럼 연령대에 따라 개인 준비물이 달라진다
- 캠핑 요리와 식재료 준비를 매번 새로 고민해야 한다
- 이동 중 여행지, 맛집, 캠핑장 주변 즐길거리까지 한 번에 판단하기 어렵다
- 기존 CLI 중심 흐름은 비개발 사용자 입장에서 반복 사용이 불편하다

## 4. 목표 사용자

v1의 목표 사용자는 아래와 같다.

- 개인 사용자
- 가족 단위 캠핑 사용자
- 소규모 동행 캠핑 사용자
- 로컬 브라우저와 로컬 앱 실행이 가능한 사용자
- 로컬 파일을 직접 수정하거나 템플릿 기반으로 관리할 수 있는 사용자

v1은 일반 대중용 서비스가 아니라 `초기 운영형 로컬 도구` 에 가깝다.

## 5. MVP 핵심 사용자 시나리오

### 시나리오 1. 날짜와 장소를 모두 정한 캠핑

1. 사용자가 장비, 동행자, 취향 데이터를 미리 저장한다
2. 사용자가 trip 파일에 날짜와 장소를 입력한다
3. UI에서 해당 trip을 선택해 분석한다
4. 결과로 장비, 개인 준비물, 요리, 이동/주변 추천을 받는다

### 시나리오 2. 날짜만 정한 캠핑

1. 사용자가 날짜만 입력한 trip 파일을 만든다
2. UI에서 해당 trip을 분석한다
3. 로컬 API가 계절성과 일반 준비 방향을 제안한다
4. 장소 미지정에 따른 한계를 함께 표시한다

### 시나리오 3. 장소만 정한 캠핑

1. 사용자가 장소 중심 trip 파일을 만든다
2. UI에서 해당 trip을 분석한다
3. 로컬 API가 장소 특성 중심 추천을 수행한다
4. 날짜 미지정에 따른 조건부 결과를 제공한다

## 6. MVP 필수 기능

### 6.1 로컬 데이터 구조

반드시 포함:

- `./.camping-data/` 디렉토리 구조 정의
- YAML 기반 입력 파일 구조 정의
- Markdown 기반 결과 파일 저장 구조 정의

목적:

- 사용자가 데이터를 반복 재사용할 수 있도록 함

### 6.2 사용자 기본 데이터 관리

반드시 포함:

- 동행자 정보 저장
- 캠핑 장비 저장
- 소모품 잔량 저장
- 출발 전 점검 항목 저장
- 여행/식사 취향 저장

비고:

- v1에서는 파일 직접 수정 방식만 있어도 충분하다
- 전체 CRUD UI는 v1 필수가 아니다

### 6.3 캠핑 요청 파일 기반 분석 실행

반드시 포함:

- `trips/*.yaml` 파일 작성 또는 선택
- trip 파일 기준 분석 실행
- 날짜만/장소만/날짜+장소 시나리오 처리

### 6.4 로컬 웹 UI

반드시 포함:

- trip 목록 선택 화면
- 분석 실행 버튼과 로딩/오류 상태 표시
- 결과 Markdown 렌더링 화면
- 결과 저장 액션

### 6.5 로컬 API

반드시 포함:

- trip 조회 API
- 분석 실행 API
- 결과 저장 API
- 입력 누락 및 충돌 검증

### 6.6 캠핑 장비 추천

반드시 포함:

- 챙겨야 할 장비 추천
- 불필요한 장비 식별
- 부족한 장비 식별
- 부족한 소모품 식별
- 사전 점검 항목 정리

### 6.7 연령대별 개인 준비물 추천

반드시 포함:

- 동행자 연령대 기반 추천
- 날씨/기온/장소 조건 반영
- 건강 특이사항과 복용약 반영

중요 원칙:

- 개인 준비물은 사용자가 직접 입력하는 목록이 아니다
- AI가 조건을 바탕으로 생성하는 추천 결과다

### 6.8 요리/식단 추천

반드시 포함:

- 사용자가 입력한 메뉴 검토
- AI 기반 메뉴 보완 추천
- 필요한 식재료 정리
- 필요한 조리도구 정리

### 6.9 이동 및 주변 추천

반드시 포함:

- 이동 중 들를 만한 장소 추천
- 이동 중 식사 후보 제안
- 캠핑장 주변 즐길거리 제안
- 캠핑장 주변 맛집/카페/마트 제안

제약:

- v1에서는 외부 데이터 자동 수집 없이도 동작 가능해야 한다
- 수동 입력 정보 또는 캐시가 있으면 품질을 높이는 수준으로 본다

### 6.10 결과 문서 생성

반드시 포함:

- 분석 결과를 Markdown으로 출력
- UI 응답과 파일 저장 모두 가능
- 결과 문서에 준비물, 체크리스트, 요리, 이동 추천이 함께 포함

### 6.11 입력 누락 처리

반드시 포함:

- 필수 입력 누락 시 보완 요청
- 제한된 정보로 분석할 때 한계 명시

예:

- 날짜와 장소가 모두 없음
- 동행자 정보 없음
- 차량 정보 부족

## 10. 품질 기준

MVP라도 아래 수준은 만족해야 한다.

- 사용자가 어떤 파일을 채워야 하는지 문서만 보고 이해 가능해야 한다
- 같은 trip 요청을 UI에서 반복 분석할 수 있어야 한다
- 추천 결과가 장비/개인 준비물/요리/이동 추천을 모두 포함해야 한다
- 날짜 또는 장소가 빠졌을 때 제한사항이 결과에 드러나야 한다
- 연령대별 개인 준비물 추천이 명확히 구분되어야 한다
- 브라우저에서 OpenAI API 키가 노출되지 않아야 한다

## 11. MVP 완료 판단 기준

아래가 가능하면 MVP가 완료된 것으로 본다.

1. 사용자가 `.camping-data/` 에 기본 데이터를 작성할 수 있다
2. 사용자가 `trips/<trip-id>.yaml` 파일을 만들 수 있다
3. UI에서 해당 trip을 선택하고 분석을 실행할 수 있다
4. 로컬 API가 AI 응답을 받아 결과를 반환할 수 있다
5. 결과 문서에 아래가 모두 포함된다

- 추천 장비 목록
- 연령대별 개인 준비물 목록
- 부족한 소모품 및 체크리스트
- 추천 식단/요리
- 이동 중 추천 장소
- 캠핑장 주변 추천 장소

## 12. 개발 우선순위

현재 MVP 구현은 완료된 상태로 본다.

### 현재 구현 완료 범위

- 문서 구조 확정
- 디렉토리 구조 확정
- 로컬 API 계약 확정
- UI 흐름 확정
- 예시 YAML/Markdown 제공
- 로컬 웹 UI와 로컬 API 구현
- 기본 분석 프롬프트와 출력 스키마 구현
- 로컬 `codex exec` 기반 분석 루프 연결

### MVP 이후 후속 우선순위

- `scripts/validate-data` 추가
- trip 템플릿 생성 스크립트 추가
- 날씨/장소 캐시 생성 보조 스크립트 추가
- export/back-up 보조 기능 추가
- 추천 결과 JSON 저장 여부 검토

## 13. 리스크와 대응

### 리스크 1. UI만 있고 파일 구조가 흐려질 수 있음

대응:

- UI는 `.camping-data/` 를 숨기지 말고 trip 단위 구조를 그대로 반영한다
- 데이터 모델과 저장 경로를 문서에 먼저 고정한다

### 리스크 2. 브라우저에 API 키를 넣는 잘못된 구현으로 흐를 수 있음

대응:

- 로컬 API를 필수 계층으로 둔다
- 환경변수와 로컬 서버 호출 구조를 체크리스트에 포함한다

### 리스크 3. 개인 준비물 추천 기준이 모호할 수 있음

대응:

- 연령대 enum과 건강 특이사항 필드를 먼저 고정한다
- 추천 결과에 이유를 함께 적는다

## 14. 현재 결정 사항

- MVP는 원격 서버 없이 진행한다
- MVP는 로컬 파일 기반 반복 분석 도구다
- 기본 실행 구조는 로컬 웹 UI와 로컬 API다
- 개인 준비물은 AI 추천 결과로 다룬다
- 핵심 분석 단위는 `trips/*.yaml` 이다
- 결과는 Markdown 문서로 저장한다
