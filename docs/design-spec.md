# 디자인 스펙

## 1. 문서 목적

이 문서는 현재 캠핑 프로젝트의 화면 디자인 기준 문서다.

목표는 아래 두 가지다.

- 최신 UI 트렌드를 무비판적으로 복제하지 않고, 현재 프로젝트에 맞는 방향으로 좁혀서 적용한다.
- 여러 개발자와 에이전트가 화면을 구현해도 비슷한 결과가 나오도록 시각 언어, 레이아웃, 상태 표현, 컴포넌트 패턴을 구체적으로 고정한다.

이 문서는 프론트엔드 화면 구성, 스타일 수정, 신규 페이지 추가, 디자인 리팩터링, Figma 없이 에이전트가 직접 화면을 구현하는 작업의 기준으로 사용한다.

## 2. 최신 참고 사례와 적용 결론

2026-03-24 기준으로 현재 화면 구조를 다시 잡을 때 참고한 공식 자료와 실제 제품 사례는 아래 5가지다.

### 2.1 Notion의 사이드바 허브 구조

- Notion은 공식 가이드에서 sidebar를 `navigation hub` 로 설명하고, 팀 공간, 공유 문서, 개인 문서를 `그룹 단위` 로 나눠 접근하게 한다.
- 이 방식은 메뉴 수가 많아져도 `지금 어디에 있는지`, `어떤 종류의 정보를 보고 있는지` 를 쉽게 이해하게 만든다.

현재 프로젝트 적용 결론:

- 메뉴는 `운영 허브`, `준비 데이터`, `관리 설정` 세 묶음으로 나누고, 준비 데이터에는 사람 관리와 차량 관리를 별도 화면으로 둔다.
- 사이드 메뉴는 페이지 이름만 나열하지 말고 각 메뉴의 역할을 짧게 설명한다.

### 2.2 Linear의 기본 뷰 중심 운영 화면

- Linear은 team pages를 sidebar의 `default views` 로 두고, 사용자가 항상 같은 위치에서 `Issues`, `Projects`, `Views` 를 오가게 만든다.
- 핵심은 화려한 카드보다 `안정적인 작업 위치`, `짧은 라벨`, `빠른 전환` 이다.

현재 프로젝트 적용 결론:

- 캠핑 계획, 장비 관리, 히스토리, 링크는 마케팅형 카드가 아니라 `항상 같은 위치에 있는 작업 뷰` 로 보여야 한다.
- 대시보드를 제외한 페이지는 bento를 과하게 쓰지 않고 `작업 헤더 + 메인 작업 영역 + 보조 패널` 구조를 우선한다.

### 2.3 GOV.UK의 가독성 중심 타입 스케일과 레이아웃

- GOV.UK Design System은 type scale을 `readability on different devices` 기준으로 반복 검증했고, 큰 화면에서도 본문이 너무 길어지지 않게 `two-thirds / one-third` 계열 레이아웃을 권장한다.
- 읽기 폭과 수직 리듬을 먼저 고정하면 화면이 커도 텍스트가 제멋대로 보이지 않는다.

현재 프로젝트 적용 결론:

- 앱 전체 최대 폭을 줄이고, 페이지 설명과 문서형 패널에는 읽기 폭 상한을 둔다.
- 본문과 폼은 sans-serif를 기본으로 쓰고, 제목만 serif로 강조한다.
- 타입 크기는 `16 / 19 / 24 / 32+` 축을 중심으로 운용한다.

### 2.4 Carbon과 WCAG 2.2의 접근성 규칙

- Carbon tabs는 tablist와 tabpanel 관계, 키보드 이동, 포커스 흐름을 기본 규칙으로 제공한다.
- WCAG 2.2는 `Focus Not Obscured`, `Consistent Help`, `Target Size` 를 강화해, 실제 서비스 화면에서 포커스와 클릭 영역을 더 엄격하게 다룬다.

현재 프로젝트 적용 결론:

- 장비 탭과 세그먼트는 시각적으로만 비슷하면 안 되고 접근성 구조까지 맞춰야 한다.
- 버튼, 탭, 카드형 클릭 영역은 실사용 기준 `44px` 안팎의 충분한 높이를 확보한다.
- 도움말, 메모, 경고 배너의 위치는 같은 종류의 화면에서 일관되게 유지한다.

### 2.5 따뜻한 로컬 도구 톤의 유지

- Notion과 Linear처럼 생산성 도구의 읽기 쉬운 질서를 참고하되, 이 프로젝트는 캠핑 도메인이므로 지나치게 차갑거나 공업적인 화면으로 가면 안 된다.
- 기존의 웜 뉴트럴 배경과 딥 그린 액센트는 유지할 가치가 있다.

현재 프로젝트 적용 결론:

- 배경은 따뜻한 종이 질감을 유지하고, 본문 가독성은 sans-serif 전환으로 보강한다.
- 강조색은 녹색 하나를 중심으로 쓰고, 경고/실패/정보는 상태 색으로만 제한한다.

## 3. 현재 프로젝트에 적용할 디자인 방향

### 3.1 방향 이름

`필드 노트 워크벤치`

### 3.2 한 줄 정의

현장 노트의 따뜻한 질감 위에, 읽기 쉬운 작업 흐름과 안정적인 운영 화면을 결합한 로컬 캠핑 관리 UI.

### 3.3 핵심 인상

- 첫눈에 메뉴 구조와 작업 순서가 읽힌다.
- 텍스트가 많아도 제목, 본문, 메타의 위계가 분명하다.
- 대시보드는 요약 화면답게 모듈형일 수 있지만, 나머지 페이지는 작업용 워크벤치처럼 차분해야 한다.
- AI는 별도 보조 영역에서만 드러나고, 사용자의 원본 입력과 절대 섞이지 않는다.
- 따뜻한 도메인 톤은 유지하되, 서비스 가능한 가독성을 최우선으로 둔다.

### 3.4 시각 키워드

- 웜 뉴트럴
- 필드 노트
- 조용한 사이드바
- 읽기 폭 제어
- serif 제목 + sans 본문
- 작업 워크벤치
- 절제된 깊이

## 4. 토큰과 시각 시스템

## 4.1 색상 시스템

기존 `apps/web/src/styles/tokens.css` 의 전역 토큰과 `apps/web/src/styles/app.css` 의 컴포넌트 규칙을 함께 기준으로 사용한다.

| 역할 | 값 | 사용 규칙 |
| --- | --- | --- |
| 배경 상단 | `#f2efe5` | 전체 페이지 배경 시작점 |
| 배경 하단 | `#d6d4cb` | 전체 페이지 배경 종료점 |
| 기본 서피스 | `rgba(255, 252, 246, 0.82)` | 일반 패널 배경 |
| 강조 서피스 | `rgba(255, 251, 245, 0.96)` | metric, 중요 카드, 내부 카드 |
| 기본 텍스트 | `#2e261f` | 제목, 본문, 버튼 텍스트 |
| 보조 텍스트 | `#6c6054` | 설명, 메타 정보, hint |
| 주 액센트 | `#2f6253` | primary action, active, 주요 태그 |
| 진한 액센트 | `#24493f` | hover, 강조 텍스트, gradient 종점 |
| 경고 | `#83571d` | 주의 텍스트 |
| 실패 | `#8a342b` | 오류 텍스트 |
| 성공 | `#2a6a4f` | 완료/정상 상태 |
| 정보 | `#315e74` | 중립 정보 배너 |

색상 규칙:

- 한 화면의 주요 액션 색은 항상 녹색 계열 한 가지로 통일한다.
- 경고/실패/정보 색은 상태 표현에만 사용하고, 일반 CTA 색으로 쓰지 않는다.
- 새로운 브랜딩 색을 임의로 추가하지 않는다.
- 배경은 따뜻한 뉴트럴을 유지하고 순수 흰색 대면적 사용을 피한다.

## 4.2 타이포그래피

### 기본 원칙

- 제목과 큰 카피는 serif로, 본문과 폼과 버튼은 sans-serif로 분리한다.
- 텍스트 계층은 `제목 / 본문 / 메타` 3단계로 단순화한다.
- 숫자/경로/코드/ID는 monospace를 사용한다.
- 타이포그래피만으로 정보 위계를 만들 수 있어야 하므로 과도한 색상 구분에 의존하지 않는다.

### 폰트 사용 규칙

| 역할 | 폰트 | 규칙 |
| --- | --- | --- |
| 페이지 인트로 제목, 페이지 제목, panel 제목 | `"Iowan Old Style", "Palatino Linotype", "Book Antiqua", Georgia, serif` | 브랜드 인상과 도메인 톤 유지 |
| 본문, 설명, 버튼, 라벨 | `"Pretendard", "Noto Sans KR", "Apple SD Gothic Neo", "Malgun Gothic", "Segoe UI", sans-serif` | 한국어 UI 가독성 우선 |
| 코드, 파일 경로, 수치 ID | `"SFMono-Regular", "JetBrains Mono", "Cascadia Code", "Fira Code", monospace` | 좁은 폭, 숫자 비교, 결과 Markdown 내부에 사용 |

### 크기 규칙

- body 기본은 `16px` 미만으로 내리지 않는다.
- 작은 메타와 eyebrow는 `12px`~`13px` 범위를 허용하지만, 버튼/폼 라벨은 `15px` 안팎을 기본값으로 둔다.
- panel title은 `1.3rem` 이상, 페이지 제목은 `2rem` 이상을 유지한다.
- 줄간은 본문 `1.6` 안팎, 설명문 `1.6~1.7` 범위를 유지한다.
- 긴 본문은 가능한 한 `60ch` 안팎의 읽기 폭 안에서 마무리한다.

### 금지 사항

- 화면 하나에 3개 이상의 폰트 계열을 섞지 않는다.
- tiny text를 장식 요소로 남발하지 않는다.
- 과도한 all caps를 한국어 본문에 쓰지 않는다.

## 4.3 간격과 라운드

현재 구현에서 반복되는 값으로 고정한다.

| 역할 | 값 |
| --- | --- |
| 큰 섹션 간격 | `24px` |
| 일반 카드 간격 | `18px` |
| 내부 그룹 간격 | `10px`~`12px` |
| panel padding | `24px` |
| page intro padding | `24px`~`26px` |
| page intro radius | `28px` |
| panel radius | `26px` |
| 일반 카드 radius | `22px` |
| 버튼/탭 radius | `20px` 또는 `999px` |
| pill radius | `999px` |

규칙:

- spacing은 `10`, `12`, `16`, `18`, `20`, `24`, `26`, `28` 축을 우선 사용한다.
- 새로운 화면을 만들 때 7px, 13px 같은 독자 값을 남발하지 않는다.

## 4.4 그림자와 블러

- 기본 그림자는 `0 24px 70px rgba(45, 40, 31, 0.12)` 축을 유지한다.
- 패널은 항상 약한 depth를 갖고, 떠 있는 배너는 이보다 강한 shadow를 사용한다.
- blur는 `16px~18px` 범위에서 사용하고, 실내용/현장용 종이 카드 느낌을 해치지 않을 정도로만 적용한다.

규칙:

- card가 너무 떠 보이는 과한 shadow는 금지한다.
- shadow 없이 border만으로 모든 층을 해결하려 하지 않는다.
- glass 효과는 필수 정보가 아닌 overlay 성격 요소에서만 제한적으로 쓴다.

## 4.5 모션

모션 규칙:

- hover/lift: `120ms`
- floating 배너 진입: `180ms`
- 일반 상태 전이: `200~300ms`
- easing은 부드럽고 짧은 ease-out 계열을 쓴다.

사용 원칙:

- 상태 변화가 있을 때만 움직인다.
- 자동 반복 애니메이션은 쓰지 않는다.
- reduced motion 환경에서는 등장 모션을 제거하거나 최소화한다.

## 5. 레이아웃 시스템

## 5.1 전체 쉘

- 전체 앱은 `min(1420px, calc(100vw - 32px))` 폭을 기본으로 한다.
- 상단 여백은 `24px`, 하단 여백은 `40px` 를 유지한다.
- 첫 인상은 항상 `sidebar + content` 조합 위에 현재 페이지 인트로가 읽히는 구조를 따른다.

## 5.2 공통 배치

### 데스크톱

- 기본 앱 레이아웃: `280px sidebar + 1fr content`
- 페이지 시작 카드: `1.25fr + 0.9fr`
- 일반 2열 페이지: `1.2fr + 0.88fr`
- 계획 페이지: `280px + 1.15fr + 0.95fr`

### 태블릿 이하

- `1240px` 이하에서는 sidebar와 주요 page grid를 1열로 접는다.
- `860px` 이하에서는 page intro와 주요 그리드를 1열로 접고 panel padding을 줄인다.
- `720px` 이하에서는 fixed toast와 metadata grid를 단일 컬럼으로 단순화한다.

## 5.3 워크벤치 규칙

- 모든 페이지를 bento처럼 만들지 않는다. `대시보드만 요약형`, 나머지는 `작업형` 이 기본이다.
- 작업형 페이지는 `헤더 -> 메인 패널 -> 보조 패널` 순서가 먼저 읽혀야 한다.
- 한 페이지 안에 `목록 / 편집 / 보조 설명` 처럼 목적이 다른 작업이 2개 이상 있으면, 세로 패널 스택보다 `page intro -> page tab -> tabpanel` 구조를 먼저 사용한다.
- page tab 순서는 특별한 이유가 없으면 `목록/현황 -> 편집/실행 -> 결과/보조` 순서로 둔다.
- 같은 성격의 작은 카드 2~4개는 그룹으로 묶되, 그룹 안에서도 제목과 설명 순서를 유지한다.
- 큰 카드 안에 작은 카드가 들어갈 때는 `panel > sub card > list item` 3단계 깊이를 넘기지 않는다.
- 카드 내부에 탭, 아코디언, 내부 카드가 동시에 겹치면 시각 피로가 커지므로 2개 이상 중첩하지 않는다.
- 프런트 상태 구조도 같은 원칙을 따른다. 페이지 draft, 선택 상태, reset version, request ref는 `useAppViewModel.tsx` 에 직접 쌓지 말고 `app/state/*` 나 feature-local hook으로 먼저 분리한다.
- 새 화면 기능이 추가될 때 `state 선언 + effect + JSX` 가 한 파일에 같이 자라기 시작하면, JSX는 `features/*`, state는 `app/state/*`, 순수 계산은 helper 모듈로 바로 쪼갠다.

## 5.4 배경 처리

- 전체 배경은 종이 같은 warm gradient를 유지한다.
- radial highlight는 좌상단과 우하단 정도만 허용한다.
- 배경 패턴은 조용해야 하며, 콘텐츠보다 존재감이 크면 안 된다.

## 5.5 프런트엔드 구현 분리 규칙

- `apps/web/src/App.tsx` 는 디자인이나 도메인 UI를 직접 그리는 위치가 아니라 `AppShell` 진입점으로만 사용한다.
- 페이지 레이아웃 책임은 `apps/web/src/pages/*Page.tsx` route entry에서 시작하고, 실제 조합은 `features/*/*PageContent.tsx` 와 하위 panel component로 위임한다.
- 새 기능이 `목록 + 편집 + 결과 + 보조 설명` 처럼 목적이 다른 영역을 한 파일에 동시에 늘리기 시작하면, 해당 영역을 별도 컴포넌트로 분리한다.
- 페이지 엔트리에서 직접 API 호출과 무거운 상태 조합을 늘리지 않는다. 입력 draft, 저장 상태, 실시간 상태 구독은 페이지 주변 훅이나 보조 모듈로 격리한다.
- 같은 페이지 안에서 major section 이 3개 이상이고 각각 독립 액션을 가지면, page entry 나 `*PageContent.tsx` 하나에 누적하지 말고 별도 panel 파일로 옮긴다.
- 페이지 탭 정의, 탭 라벨, `sessionStorage` 기반 UI 복원 규칙은 `apps/web/src/app/ui-state.ts` 에 모으고, view model 안에 다시 정의하지 않는다.
- 상위 파일에서 시각 규칙과 도메인 로직이 다시 뒤섞이지 않도록, `App.tsx` 와 `AppShell.tsx` 는 `apiClient` 직접 호출 금지 위치로 본다.
- 전환 단계의 `useAppViewModel.tsx` 같은 adapter 파일은 새 화면 책임을 더 모으는 위치가 아니라, 분리 작업이 끝날 때까지 줄여 나가는 예외 파일로 취급한다.
- 공통 변환 로직은 `app/common-formatters.ts`, draft 기본값과 저장 변환은 `app/view-model-drafts.ts`, 도메인 선택/상태 helper 는 `app/*-helpers.ts` 로 먼저 분리한다.
- state 변화에 따라 자동으로 따라오는 저장/복원/잠금 로직은 `app/effects/*` 로 분리하고, CRUD 핸들러는 `features/*/actions.ts` 같은 도메인 action 모듈로 뺀다.
- 테스트도 구조를 따른다. 시나리오 본문은 `App.test.tsx`, fetch/EventSource mock 과 기본 상태 팩토리는 `src/test/app-test-helpers.tsx`, `src/test/mock-state.ts` 로 분리한다.
- 전역 스타일 entry는 `styles/index.css` 하나만 `main.tsx` 에서 불러오고, 토큰과 반응형 media query 는 `tokens.css`, `responsive.css` 로 나눠 유지한다.

## 6. 컴포넌트 스펙

## 6.1 Page Intro

- 각 페이지의 시작점은 `현재 위치`, `핵심 문장`, `보조 설명`, `현재 상태 요약` 을 포함하는 page intro 카드로 구성한다.
- 페이지 인트로 오른쪽 메타 카드에는 현재 선택 항목, 경고 수, 링크 그룹 수처럼 `문맥 복원` 에 필요한 정보만 둔다.
- 파일 경로와 생성 규칙처럼 설명성 성격이 강한 정보는 page intro에 두지 않고 `보조 설명` 화면으로 분리한다.
- page intro는 마케팅 배너처럼 화려하면 안 되고 운영형 도구의 시작 카드처럼 보여야 한다.

## 6.2 Panel

- 모든 주요 섹션은 panel을 기준 단위로 쓴다.
- panel 안에는 `eyebrow`, `title`, `copy`, `actions` 의 순서를 우선한다.
- copy는 한 문단 2~3문장 이내로 유지한다.

## 6.3 Metric Card

- 숫자 KPI는 2x2 grid 또는 1열 반복 카드로 보여준다.
- 숫자 위에는 설명, 아래에는 추가 설명을 두지 않고 상단 label + 하단 strong 조합을 유지한다.
- 지표 수는 한 그룹당 최대 4개를 권장한다.

## 6.4 리스트 카드와 아코디언 카드

- 장비, 계획, 히스토리 같은 반복 목록은 `카드형 리스트` 를 사용한다.
- 카드 hover는 `translateY(-1px)` 정도의 약한 lift만 허용한다.
- active 상태는 border와 background tint로 표현한다.
- 아코디언을 쓸 때는 summary 줄 자체가 카드 헤더 역할을 하도록 하고, detail은 상단 border로만 분리한다.
- 상위 그룹과 하위 항목이 함께 있는 목록은 `그룹 헤더 > 하위 리스트 > 상세 패널` 위계가 한눈에 읽혀야 하며, 하위 항목은 들여쓰기나 보조 서피스로 한 단계 낮춰 보인다.

## 6.5 탭과 세그먼트

- 섹션 전환은 pills가 아니라 `둥근 세그먼트/탭` 을 사용한다.
- 활성 탭은 항상 녹색 gradient fill을 사용한다.
- 비활성 탭은 반투명 서피스 위에 두고 hover에서만 tint를 준다.

## 6.6 폼

- 폼은 입력 밀도가 높아도 시각적으로 숨 쉴 공간이 있어야 한다.
- label, help text, field, inline action의 순서를 유지한다.
- 한 줄에 너무 많은 입력을 넣지 않는다. 특히 모바일에서 1열로 자연스럽게 내려와야 한다.
- 필수/권장/AI 제안 입력을 색만으로 구분하지 않는다.

## 6.7 상태 배너

- status banner는 정보, 경고, 오류, 성공 4종류만 사용한다.
- floating banner는 화면 상단 우측에 두되 모바일에서는 전체 폭에 가깝게 확장한다.
- 설명과 선택적 bullet list를 허용하지만, 배너 안에 버튼을 과도하게 넣지 않는다.

## 6.8 AI 보조 패널

- AI 보조는 채팅 앱처럼 보이지 않아야 한다.
- `입력 박스 + 응답 카드 + 제안 액션 카드` 구조를 우선한다.
- AI가 자동으로 바꾼 것처럼 보이는 표현은 금지한다.
- 응답에는 `제안`, `이유`, `적용 전 사용자가 확인해야 할 항목` 이 드러나야 한다.
- AI 제안과 실제 저장/적용 액션은 반드시 시각적으로 분리한다.

## 6.9 결과 Markdown 패널

- 분석 결과는 문서처럼 읽히되, 페이지 전체와 단절되지 않게 같은 서피스 언어를 사용한다.
- 결과 패널 상단에는 섹션 체크 목록, 상태 칩, `선택 수집` 과 `전체 실행` 액션을 함께 둔다.
- 섹션 카드는 체크박스, 섹션 번호/제목, 마지막 수집 시각, `재수집` 버튼을 한 행에서 읽을 수 있게 구성한다.
- `수집 완료`, `대기 중`, `수집 중`, `실패`, `중단` 상태는 pill chip 으로 분명히 구분한다.
- Markdown 제목과 본문 간격은 기존 CSS 축을 유지한다.
- inline code는 액센트 tint 배경을 사용하고 과도하게 강조하지 않는다.
- 백그라운드 분석 중에는 정보 배너와 `분석 중...` 상태를 함께 보여 줘야 한다.
- 분석 실패 또는 중단 시에는 기존 Markdown을 지우지 않고 상태 배너만 갱신한다.

## 6.10 표와 데이터 밀도

향후 표가 필요한 경우 아래 규칙을 따른다.

- 표 title과 description을 명시한다.
- global action은 table toolbar에만 둔다.
- search는 title 아래 왼쪽, action은 오른쪽 배치 원칙을 따른다.
- pagination은 항상 표 하단에 붙인다.
- 행 hover는 항상 제공한다.
- 정렬 가능한 컬럼은 hover와 active 아이콘으로만 표시하고, 모든 컬럼에 아이콘을 상시 노출하지 않는다.
- 툴바 아이콘/버튼은 5개 이내를 권장한다.

## 7. 화면별 구성 규칙

## 7.1 대시보드

- `page intro`
- 페이지 탭 `운영 요약`, `빠른 실행`, `최근 기록`
- `운영 요약` 탭에는 `운영 지표 2x2 metric grid`, `예정 계획`, `점검 경고` 를 둔다
- `빠른 실행` 탭에는 다음 작업으로 넘어가는 CTA와 현재 선택 상태 요약만 둔다
- `최근 기록` 탭에는 최근 히스토리와 외부 링크 요약을 묶는다

대시보드는 가장 요약형이어도 되는 화면이지만, 카드 수보다 `오늘 무엇을 해야 하는지` 가 먼저 보여야 한다.

## 7.2 장비 관리

- 상단에 섹션 탭
- 섹션 탭 아래에 페이지 탭 `장비 목록`, `상세 작업`
- `장비 목록` 탭에는 카테고리 아코디언과 아이템 상세 패널을 둔다
- `상세 작업` 탭은 상단 요약 스트립과 `작업 요약`, `항목 추가` 내부 탭으로 나눈다
- 반복 장비 메타데이터는 별도 `metadata card` 로 격리
- 반복 장비 요약 행에는 메타데이터 작업 상태를 짧은 배지로 함께 표시한다
- 메타데이터 재수집이 `queued` 또는 `running` 이면 버튼 라벨은 `메타데이터 수집 중...` 으로 유지한다
- 기존 메타데이터가 있는 상태에서 재수집 중이면 카드는 이전 메타데이터를 유지한 채 `백그라운드 수집 중` 안내만 추가한다
- 여러 반복 장비가 동시에 수집 중이어도 각 상세 카드와 요약 행이 서로 독립적으로 상태를 보여야 한다
- 목록 편집보다 `상태 파악 -> 선택 -> 수정` 흐름이 먼저 읽혀야 한다
- 카테고리 그룹과 장비 상세 패널은 모두 기본 접힘 상태로 시작한다.
- 카테고리는 장비보다 한 단계 더 무겁게 보이고, 카테고리 안 항목은 보조 서피스와 계층 레일로 묶여 보여야 한다.

## 7.3 캠핑 계획

- 페이지 탭 `계획 목록`, `원본 입력`, `AI·결과` 를 기본 구조로 사용한다.
- `계획 목록` 탭에는 계획 카드 리스트와 `새 계획 작성` 진입점을 둔다.
- `원본 입력` 탭에는 계획 폼과 저장/분석/아카이브 액션을 둔다.
- `AI·결과` 탭에는 상단 요약 카드와 내부 상세 탭 `분석 결과`, `AI 보조`, `누적 학습` 을 둔다.
- 동행자와 차량은 자유 입력 대신 `선택 블록 + 선택 요약 카드` 구조를 우선한다.
- 사람/차량 CRUD는 이 화면 안에 섞지 않고 별도 관리 화면으로 이동시키는 버튼만 둔다.
- 내부 상세 탭은 `분석 결과`, `AI 보조`, `누적 학습` 3개로 고정하고 기본 활성 탭은 `분석 결과` 로 둔다.
- 결과 Markdown은 AI 응답보다 무겁고 정적인 panel로 보여야 하며, `AI·결과` 탭 안에서 가장 먼저 읽히게 둔다.
- 사용자의 원본 입력과 AI 제안은 반드시 다른 박스에 둔다.
- 분석이 `queued` 또는 `running` 이면 분석 버튼은 비활성화하고 `분석 중...` 라벨을 유지한다.
- 분석 중에는 삭제와 히스토리 이동 버튼도 함께 비활성화해 충돌 가능성을 먼저 차단한다.

## 7.4 캠핑 히스토리

- 페이지 탭 `히스토리 목록`, `상세 보기` 를 기본 구조로 사용한다.
- `히스토리 목록` 탭에는 기록 카드 리스트를, `상세 보기` 탭에는 선택된 기록 요약과 내부 상세 탭을 둔다.
- 목록과 상세 모두 `누구와 갔는지`, `어떤 차량을 썼는지` 가 바로 읽혀야 한다.
- 상세 패널 상단에는 당시 동행자/차량 스냅샷과 현재 학습 상태를 요약 카드로 노출한다.
- 상세 탭은 `요약`, `기록/결과`, `후기 작성`, `학습` 4개로 고정한다.
- `요약` 탭에는 기본 정보와 스냅샷 요약을 두고, `후기 작성` 탭에는 회고 입력 폼 전체를 둔다.
- `학습` 탭에는 이번 캠핑에서 AI가 배운 점과 전역 개인화 학습 요약을 함께 둔다.
- `기록/결과` 탭에는 회고 엔트리 목록, 저장된 결과 Markdown 다시 열기, 메모 저장/삭제 액션을 묶는다.
- 회고 입력 폼은 단순 메모 textarea 하나가 아니라 구조화 필드와 자유 후기를 함께 담는 스택 레이아웃을 사용한다.
- 개인화 학습이 `queued` 또는 `running` 이면 결과 카드를 비우지 말고, 마지막 성공 결과 위에 상태 배너만 덧붙인다.
- 과거 기록은 차분하고 아카이브 느낌을 주되, 비활성화된 화면처럼 보여서는 안 된다

## 7.5 외부 링크

- 페이지 탭 `링크 목록`, `새 링크` 를 사용한다.
- 카테고리 그룹을 분명히 나누고, 각 그룹을 카드로 묶는다.
- 링크는 북마크처럼 빠르게 읽혀야 하므로 제목, 설명, URL, 메모 순서를 짧게 유지한다.
- 외부 이동 버튼은 항상 같은 위치에 둔다.

## 7.6 관리 설정

- 관리성 화면이지만 차갑고 기술적인 화면으로 보이지 않게 한다.
- 페이지 탭 `카테고리 목록`, `보조 작업` 을 기본 구조로 사용한다.
- `카테고리 목록` 탭에는 섹션 아코디언과 카테고리 편집 카드를 둔다.
- `보조 작업` 탭은 상단 요약 스트립과 `새 카테고리`, `관리 원칙`, `로컬 백업` 내부 탭으로 나눈다.
- 카테고리 목록과 수정 입력을 한눈에 묶고, 삭제 불가 사유는 배너/보조 텍스트로 설명한다.
- 카테고리 편집 카드는 기본 접힘 상태로 두고, 펼쳤을 때만 수정 입력과 액션을 노출한다.

## 7.7 사람 관리와 차량 관리

- 사람 관리는 페이지 탭 `사람 목록`, `프로필 편집` 을 사용한다.
- 차량 관리는 페이지 탭 `차량 목록`, `차량 편집` 을 사용한다.
- 계획 화면에서 다시 입력하지 않아도 되게, 이름/설명/건강 메모/적재량 같은 핵심 요약이 목록에서 먼저 읽혀야 한다.

## 7.8 보조 설명

- 페이지 탭 `파일 안내`, `운영 메모` 를 사용한다.
- 경로, 생성 규칙, 파일 역할 같은 설명성 정보는 이 화면으로만 모은다.
- 코드성 텍스트가 필요하면 monospace를 쓰되, 사용자 문장 안에 raw backtick을 그대로 노출하지 않는다.

## 8. 에이전트 구현 규칙

- 화면 작업 전에는 반드시 이 문서를 읽고, 어떤 화면 패턴을 따를지 먼저 정한다.
- 새 UI를 만들 때는 먼저 `apps/web/src/styles/tokens.css` 토큰과 `apps/web/src/styles/app.css` 컴포넌트 언어를 재사용하고, 정말 필요한 경우에만 토큰을 확장한다.
- 색상, 간격, radius, shadow를 하드코딩하지 않는다. 기존 축과 의미를 우선 맞춘다.
- 외부 폰트, 아이콘 패키지, 과도한 애니메이션 라이브러리를 기본값으로 추가하지 않는다.
- AI 패널은 `보조 도구` 로 보여야지 `대화 앱` 처럼 보이면 안 된다.
- 본문을 serif로 밀어붙이지 않는다. 한국어 본문과 폼은 sans-serif를 기본으로 한다.
- 사이드 메뉴는 그룹 제목, 메뉴 설명, 상태 메타가 함께 읽히는 구조를 유지한다.
- 데스크톱 사이드 메뉴는 독립 스크롤 영역으로 유지하고, 본문 스크롤과 체감 충돌이 없게 `overscroll-behavior` 와 고정 높이 규칙을 함께 적용한다.
- 사용자가 별도로 지시하지 않아도 purple 계열 미래지향 UI, pure white 배경, glass 카드 남발, dark mode 우선 디자인을 만들지 않는다.
- 이미지가 꼭 필요하지 않은 운영형 화면에서 stock photo hero를 넣지 않는다.
- 새 컴포넌트는 데스크톱, `1240px` 이하, `860px` 이하, `720px` 이하 상태를 함께 고려한다.

## 9. 금지 패턴

- 보라색 중심의 AI SaaS 템플릿 룩
- full dark mode를 기본 전제로 한 화면
- 과도한 3D 오브젝트, 자동 반복 애니메이션, 장식용 파티클
- 테이블과 카드에 진한 border를 여러 겹 두르는 방식
- 한 화면에 3개 이상의 강조색 사용
- AI 제안과 사용자 원본 입력을 섞어 보여주는 배치
- 버튼, 탭, 링크가 모두 서로 다른 스타일 언어를 갖는 상태
- 모바일에서 터치 타깃이 너무 작아지는 조밀한 toolbar

## 10. 구현 체크리스트

프론트엔드 화면 작업 전에 아래를 점검한다.

1. 이 화면이 현재 메뉴 그룹과 페이지 체계 중 어디에 속하는지 정했는가
2. 이 화면이 `요약형` 인지 `작업형` 인지 먼저 정했는가
3. 사이드 메뉴와 페이지 헤더에서 현재 위치가 한 번 더 복원되는가
4. 본문, 버튼, 폼 라벨이 최소 가독성 크기를 지키는가
5. serif 제목과 sans 본문이 일관되게 유지되는가
6. hover/focus/active/success/error 상태가 모두 정의돼 있는가
7. AI 관련 요소가 자동 적용처럼 오해되지 않는가
8. `1240px`, `860px`, `720px` 이하에서 정보 우선순위가 유지되는가
9. 그림자, blur, gradient가 내용보다 튀지 않는가
