# 디자인 스펙

## 1. 문서 목적

이 문서는 현재 캠핑 프로젝트의 화면 디자인 기준 문서다.

목표는 아래 두 가지다.

- 최신 UI 트렌드를 무비판적으로 복제하지 않고, 현재 프로젝트에 맞는 방향으로 좁혀서 적용한다.
- 여러 개발자와 에이전트가 화면을 구현해도 비슷한 결과가 나오도록 시각 언어, 레이아웃, 상태 표현, 컴포넌트 패턴을 구체적으로 고정한다.

이 문서는 프론트엔드 화면 구성, 스타일 수정, 신규 페이지 추가, 디자인 리팩터링, Figma 없이 에이전트가 직접 화면을 구현하는 작업의 기준으로 사용한다.

## 2. 최신 트렌드 조사 요약

최근 2025~2026 자료를 기준으로 현재 프로젝트에 유효한 베스트 프랙틱스를 5개로 정리하면 아래와 같다.

### 2.1 모듈형 Bento 레이아웃

- 최근 운영형 대시보드와 SaaS 화면은 긴 단일 컬럼보다 `목적이 다른 카드가 모듈형으로 배치된 bento 레이아웃` 을 선호한다.
- 이유는 복잡한 정보를 한 번에 보여주면서도 `빠른 스캔`, `카드 단위 우선순위`, `모바일 적응성` 을 확보하기 쉽기 때문이다.
- 현재 프로젝트처럼 대시보드, 장비 상태, 히스토리, AI 제안, 링크 현황이 동시에 존재하는 로컬 운영 도구와 잘 맞는다.

현재 프로젝트 적용 결론:

- 첫 화면과 운영형 목록 화면은 `모듈형 카드 배열` 을 기본으로 사용한다.
- 카드 크기로 중요도를 조정하고, 탭이나 아코디언은 카드 안에서 보조적으로만 쓴다.

### 2.2 Warm Minimalism과 단일 액센트

- 2025 UI는 차갑고 하얀 플랫 화면보다 `따뜻한 중성 배경`, `여유 있는 여백`, `단일 액센트 컬러` 를 선호한다.
- 특히 오래 머무는 생산성 도구와 운영형 화면에서 따뜻한 베이지, 크림, 웜그레이 계열이 피로를 줄이고 제품의 인간적인 톤을 강화한다.
- 액센트 컬러는 여러 개를 섞기보다 하나를 강하게 반복하는 편이 학습 가능성과 브랜드 인지 측면에서 유리하다.

현재 프로젝트 적용 결론:

- 기존의 웜 베이지 배경과 딥 그린 액센트를 유지한다.
- 새로운 화면을 만들 때 블루, 퍼플, 네온 계열을 도입하지 않는다.
- 강조 색은 현재의 녹색 계열 하나를 중심으로 쓰고, 경고/실패/정보는 상태 색으로만 제한한다.

### 2.3 절제된 깊이와 목적 있는 모션

- 최신 UI는 무거운 skeuomorphism보다 `얕은 그림자`, `은은한 블러`, `레이어 차이`, `짧고 목적 있는 모션` 으로 깊이를 만든다.
- 모션은 장식이 아니라 `상태 전이`, `주의 유도`, `성공/실패 피드백`, `hover/tap 감각` 을 위해 사용해야 한다.
- 특히 카드형 운영 UI에서는 미세한 lift, blur, shadow만으로도 현대적인 밀도를 만들 수 있다.

현재 프로젝트 적용 결론:

- 카드와 패널은 종이 위에 얹힌 도구처럼 보여야 한다.
- hover, 탭 전환, 배너 등장, 카드 펼침은 짧은 모션을 허용하지만, 큰 패럴랙스나 계속 움직이는 장식 애니메이션은 금지한다.
- blur는 `hero`, `floating status`, `overlay`, `AI 관련 부유 카드` 정도에만 제한적으로 사용한다.

### 2.4 접근성은 트렌드가 아니라 기본값

- WCAG 2.2 기준에서 `focus가 가려지지 않아야 한다`, `터치 타깃은 최소 24x24 CSS px`, `도움말 위치는 일관돼야 한다` 같은 항목이 강화됐다.
- 최신 제품은 접근성을 별도 체크박스가 아니라 기본 설계 원칙으로 다룬다.
- 데이터가 많은 운영형 UI일수록 명확한 대비, 넉넉한 타깃, 반복되는 도움 위치, 키보드 포커스 표시가 중요하다.

현재 프로젝트 적용 결론:

- 모든 버튼, 탭, 카드형 클릭 영역, 배너 닫기 버튼은 포커스가 또렷하고 가려지지 않아야 한다.
- 작은 아이콘 버튼만으로 의미를 전달하지 않는다.
- 색만으로 상태를 구분하지 말고 텍스트, 아이콘, 라벨을 함께 제공한다.

### 2.5 AI 시대에는 디자인 시스템 문서가 구현 품질을 좌우한다

- 2025~2026 흐름에서는 화면을 사람이 직접만 구현하지 않고, 에이전트가 코드까지 만드는 경우가 많다.
- 이때 결과물 편차를 줄이려면 “예쁘게” 같은 추상 표현보다 `어떤 토큰을 쓰는지`, `카드를 어디에 두는지`, `절대 하지 말아야 할 것은 무엇인지` 를 명시한 설계 문서가 필요하다.
- Figma와 AI 대시보드 도구들도 `existing components`, `tokens`, `real data states`, `actionable rules` 를 강조한다.

현재 프로젝트 적용 결론:

- 이 문서는 디자인 감상문이 아니라 구현 규칙 문서여야 한다.
- AGENTS/CLAUDE 문서에서 이 문서를 직접 참조하게 만들어, 사용자가 별도 요청하지 않아도 프론트엔드 결과물에 반영되도록 한다.

## 3. 현재 프로젝트에 적용할 디자인 방향

### 3.1 방향 이름

`필드 노트 대시보드`

### 3.2 한 줄 정의

야외 현장 노트의 따뜻한 질감과 운영형 로컬 대시보드의 명료함을 결합한 캠핑 관리자 UI.

### 3.3 핵심 인상

- 차갑지 않고 오래 봐도 피로가 적다.
- 카드가 많아도 산만하지 않고 우선순위가 읽힌다.
- AI 기능은 과장된 챗봇 UI가 아니라 `보조 패널` 로 드러난다.
- 데이터와 문서가 중심이며, 이미지 의존도가 낮다.
- 세련되지만 캠핑 도메인의 자연스러운 온기를 잃지 않는다.

### 3.4 시각 키워드

- 웜 뉴트럴
- 종이 같은 배경
- 딥 포레스트 그린 액센트
- 레이어드 패널
- 에디토리얼 타이포그래피
- 모듈형 정보 카드
- 절제된 블러와 그림자

## 4. 토큰과 시각 시스템

## 4.1 색상 시스템

기존 `apps/web/src/styles/app.css` 의 토큰을 기준으로 사용한다.

| 역할 | 값 | 사용 규칙 |
| --- | --- | --- |
| 배경 상단 | `#f2efe5` | 전체 페이지 배경 시작점 |
| 배경 하단 | `#d6d4cb` | 전체 페이지 배경 종료점 |
| 기본 서피스 | `rgba(255, 252, 246, 0.82)` | 일반 패널 배경 |
| 강조 서피스 | `rgba(255, 251, 245, 0.96)` | metric, 중요 카드, 내부 카드 |
| 기본 텍스트 | `#2e261f` | 제목, 본문, 버튼 텍스트 |
| 보조 텍스트 | `#6c6054` | 설명, 메타 정보, hint |
| 주 액센트 | `#2f6253` | primary action, active, 주요 태그 |
| 진한 액센트 | `#24493f` | hover, 강조 텍스트, gradient 종점 |
| 경고 | `#83571d` | 주의 텍스트 |
| 실패 | `#8a342b` | 오류 텍스트 |
| 성공 | `#2a6a4f` | 완료/정상 상태 |
| 정보 | `#315e74` | 중립 정보 배너 |

색상 규칙:

- 한 화면의 주요 액션 색은 항상 녹색 계열 한 가지로 통일한다.
- 경고/실패/정보 색은 상태 표현에만 사용하고, 일반 CTA 색으로 쓰지 않는다.
- 새로운 브랜딩 색을 임의로 추가하지 않는다.
- 배경은 따뜻한 뉴트럴을 유지하고 순수 흰색 대면적 사용을 피한다.

## 4.2 타이포그래피

### 기본 원칙

- 제목과 큰 카피는 기존 serif 스택을 유지한다.
- 본문도 현재 구현과 충돌하지 않도록 같은 계열을 기본으로 쓰되, 숫자/경로/코드/ID는 monospace를 사용한다.
- 타이포그래피만으로 정보 위계를 만들 수 있어야 하므로 과도한 색상 구분에 의존하지 않는다.

### 폰트 사용 규칙

| 역할 | 폰트 | 규칙 |
| --- | --- | --- |
| hero 제목, 페이지 제목, panel 제목 | `"Iowan Old Style", "Palatino Linotype", "Book Antiqua", Georgia, serif` | 브랜드 인상과 도메인 톤 유지 |
| 본문, 설명, 버튼, 라벨 | 위 serif 스택 유지 | 현재 구현과 일관성 유지 |
| 코드, 파일 경로, 수치 ID | `"SFMono-Regular", "JetBrains Mono", "Cascadia Code", "Fira Code", monospace` | 좁은 폭, 숫자 비교, 결과 Markdown 내부에 사용 |

### 크기 규칙

- body 기본은 `16px` 미만으로 내리지 않는다.
- 설명 문구는 `14px` 까지는 허용하지만, 핵심 조작 라벨은 `14px` 미만 금지.
- panel title은 `1.35rem` 이상, hero h1은 `2.1rem` 이상을 유지한다.
- 줄간은 본문 `1.5~1.6`, 긴 설명문 `1.55~1.7` 범위를 유지한다.

### 금지 사항

- 화면 하나에 3개 이상의 폰트 계열을 섞지 않는다.
- tiny text를 장식 요소로 남발하지 않는다.
- 과도한 all caps를 한국어 본문에 쓰지 않는다.

## 4.3 간격과 라운드

현재 구현에서 반복되는 값으로 고정한다.

| 역할 | 값 |
| --- | --- |
| 큰 섹션 간격 | `24px` |
| 일반 카드 간격 | `16px` |
| 내부 그룹 간격 | `10px`~`12px` |
| panel padding | `22px` |
| hero padding | `24px` |
| hero radius | `30px` |
| panel radius | `28px` |
| 일반 카드 radius | `22px` |
| 버튼/탭 radius | `18px` |
| pill radius | `999px` |

규칙:

- spacing은 `10`, `12`, `16`, `18`, `22`, `24`, `28`, `30` 축을 우선 사용한다.
- 새로운 화면을 만들 때 7px, 13px 같은 독자 값을 남발하지 않는다.

## 4.4 그림자와 블러

- 기본 그림자는 `0 24px 70px rgba(45, 40, 31, 0.12)` 축을 유지한다.
- 패널은 항상 약한 depth를 갖고, 떠 있는 배너는 이보다 강한 shadow를 사용한다.
- blur는 `16px~18px` 범위에서 사용하고, 실내용/현장용 종이 카드 느낌을 해치지 않을 정도로만 적용한다.

규칙:

- card가 너무 떠 보이는 과한 shadow는 금지한다.
- shadow 없이 border만으로 모든 층을 해결하려 하지 않는다.
- glass 효과는 필수 정보가 아닌 overlay 성격 요소에서만 제한적으로 쓴다.

## 4.5 모션

모션 규칙:

- hover/lift: `120ms`
- floating 배너 진입: `180ms`
- 일반 상태 전이: `200~300ms`
- easing은 부드럽고 짧은 ease-out 계열을 쓴다.

사용 원칙:

- 상태 변화가 있을 때만 움직인다.
- 자동 반복 애니메이션은 쓰지 않는다.
- reduced motion 환경에서는 등장 모션을 제거하거나 최소화한다.

## 5. 레이아웃 시스템

## 5.1 전체 쉘

- 전체 앱은 `min(1500px, calc(100vw - 32px))` 폭을 기본으로 한다.
- 상단 여백은 `24px`, 하단 여백은 `40px` 를 유지한다.
- 첫 인상은 항상 `hero + content layout` 조합을 따른다.

## 5.2 공통 배치

### 데스크톱

- 기본 앱 레이아웃: `260px sidebar + 1fr content`
- 일반 2열 페이지: `1.2fr + 0.9fr`
- 계획 페이지: `300px + 1.1fr + 1fr`

### 태블릿 이하

- `1240px` 이하에서는 sidebar와 주요 page grid를 1열로 접는다.
- `860px` 이하에서는 hero도 1열로 접고 panel padding을 줄인다.
- `720px` 이하에서는 fixed toast와 metadata grid를 단일 컬럼으로 단순화한다.

## 5.3 Bento 규칙

- 한 화면에서 가장 중요한 정보는 가장 큰 카드에 둔다.
- 같은 성격의 작은 카드 2~4개는 group으로 묶고, gap은 항상 동일하게 유지한다.
- 큰 카드 안에 작은 카드가 들어갈 때는 `panel > sub card > list item` 3단계 깊이를 넘기지 않는다.
- 카드 내부에 또 탭, 아코디언, 내부 카드가 동시에 중첩되면 시각 피로가 크므로 2개 이상 겹치지 않는다.

## 5.4 배경 처리

- 전체 배경은 종이 같은 warm gradient를 유지한다.
- radial highlight는 좌상단과 우하단 정도만 허용한다.
- 배경 패턴은 조용해야 하며, 콘텐츠보다 존재감이 크면 안 된다.

## 6. 컴포넌트 스펙

## 6.1 Hero

- 페이지 시작점은 반드시 `현재 위치`, `핵심 문장`, `보조 설명`, `현재 상태나 경로 요약` 을 포함한다.
- hero 오른쪽 메타 카드에는 시스템 상태, 현재 선택 항목, 결과 경로, 마지막 실행 정보처럼 `문맥 복원` 에 필요한 정보만 둔다.
- hero는 마케팅 배너처럼 화려하면 안 되고 운영형 도구의 시작 카드처럼 보여야 한다.

## 6.2 Panel

- 모든 주요 섹션은 panel을 기준 단위로 쓴다.
- panel 안에는 `eyebrow`, `title`, `copy`, `actions` 의 순서를 우선한다.
- copy는 한 문단 2~3문장 이내로 유지한다.

## 6.3 Metric Card

- 숫자 KPI는 2x2 grid 또는 1열 반복 카드로 보여준다.
- 숫자 위에는 설명, 아래에는 추가 설명을 두지 않고 상단 label + 하단 strong 조합을 유지한다.
- 지표 수는 한 그룹당 최대 4개를 권장한다.

## 6.4 리스트 카드와 아코디언 카드

- 장비, 계획, 히스토리 같은 반복 목록은 `카드형 리스트` 를 사용한다.
- 카드 hover는 `translateY(-1px)` 정도의 약한 lift만 허용한다.
- active 상태는 border와 background tint로 표현한다.
- 아코디언을 쓸 때는 summary 줄 자체가 카드 헤더 역할을 하도록 하고, detail은 상단 border로만 분리한다.

## 6.5 탭과 세그먼트

- 섹션 전환은 pills가 아니라 `둥근 세그먼트/탭` 을 사용한다.
- 활성 탭은 항상 녹색 gradient fill을 사용한다.
- 비활성 탭은 반투명 서피스 위에 두고 hover에서만 tint를 준다.

## 6.6 폼

- 폼은 입력 밀도가 높아도 시각적으로 숨 쉴 공간이 있어야 한다.
- label, help text, field, inline action의 순서를 유지한다.
- 한 줄에 너무 많은 입력을 넣지 않는다. 특히 모바일에서 1열로 자연스럽게 내려와야 한다.
- 필수/권장/AI 제안 입력을 색만으로 구분하지 않는다.

## 6.7 상태 배너

- status banner는 정보, 경고, 오류, 성공 4종류만 사용한다.
- floating banner는 화면 상단 우측에 두되 모바일에서는 전체 폭에 가깝게 확장한다.
- 설명과 선택적 bullet list를 허용하지만, 배너 안에 버튼을 과도하게 넣지 않는다.

## 6.8 AI 보조 패널

- AI 보조는 채팅 앱처럼 보이지 않아야 한다.
- `입력 박스 + 응답 카드 + 제안 액션 카드` 구조를 우선한다.
- AI가 자동으로 바꾼 것처럼 보이는 표현은 금지한다.
- 응답에는 `제안`, `이유`, `적용 전 사용자가 확인해야 할 항목` 이 드러나야 한다.
- AI 제안과 실제 저장/적용 액션은 반드시 시각적으로 분리한다.

## 6.9 결과 Markdown 패널

- 분석 결과는 문서처럼 읽히되, 페이지 전체와 단절되지 않게 같은 서피스 언어를 사용한다.
- Markdown 제목과 본문 간격은 기존 CSS 축을 유지한다.
- inline code는 액센트 tint 배경을 사용하고 과도하게 강조하지 않는다.

## 6.10 표와 데이터 밀도

향후 표가 필요한 경우 아래 규칙을 따른다.

- 표 title과 description을 명시한다.
- global action은 table toolbar에만 둔다.
- search는 title 아래 왼쪽, action은 오른쪽 배치 원칙을 따른다.
- pagination은 항상 표 하단에 붙인다.
- 행 hover는 항상 제공한다.
- 정렬 가능한 컬럼은 hover와 active 아이콘으로만 표시하고, 모든 컬럼에 아이콘을 상시 노출하지 않는다.
- 툴바 아이콘/버튼은 5개 이내를 권장한다.

## 7. 화면별 구성 규칙

## 7.1 대시보드

- `hero`
- `운영 지표 2x2 metric grid`
- `예정 계획`, `최근 히스토리` 2열 카드
- `점검 경고`, `빠른 이동`, `외부 링크 요약` 보조 카드

대시보드는 가장 bento스럽게 보여도 되는 화면이다.

## 7.2 장비 관리

- 상단에 섹션 탭
- 탭 아래에 카테고리/목록/상세 확장 카드
- 반복 장비 메타데이터는 별도 `metadata card` 로 격리
- 목록 편집보다 `상태 파악 -> 선택 -> 수정` 흐름이 먼저 읽혀야 한다

## 7.3 캠핑 계획

- 데스크톱은 `계획 목록 / 계획 폼 / AI 보조+결과` 3열을 기본으로 한다.
- 폼은 가장 넓은 작업 영역으로 보이게 하고, AI는 오른쪽 보조 열에 둔다.
- 결과 Markdown은 AI 응답보다 무겁고 정적인 panel로 보여야 한다.
- 사용자의 원본 입력과 AI 제안은 반드시 다른 박스에 둔다.

## 7.4 캠핑 히스토리

- 좌측은 기록 목록, 우측은 선택된 기록 요약과 결과 다시 열기 패널
- 과거 기록은 차분하고 아카이브 느낌을 주되, 비활성화된 화면처럼 보여서는 안 된다

## 7.5 외부 링크

- 카테고리 그룹을 분명히 나누고, 각 그룹을 카드로 묶는다.
- 링크는 북마크처럼 빠르게 읽혀야 하므로 제목, 설명, URL, 메모 순서를 짧게 유지한다.
- 외부 이동 버튼은 항상 같은 위치에 둔다.

## 7.6 관리 설정

- 관리성 화면이지만 차갑고 기술적인 화면으로 보이지 않게 한다.
- 카테고리 목록과 수정 입력을 한눈에 묶고, 삭제 불가 사유는 배너/보조 텍스트로 설명한다.

## 8. 에이전트 구현 규칙

- 화면 작업 전에는 반드시 이 문서를 읽고, 어떤 화면 패턴을 따를지 먼저 정한다.
- 새 UI를 만들 때는 먼저 기존 `apps/web/src/styles/app.css` 토큰과 컴포넌트 언어를 재사용하고, 정말 필요한 경우에만 토큰을 확장한다.
- 색상, 간격, radius, shadow를 하드코딩하지 않는다. 기존 축과 의미를 우선 맞춘다.
- 외부 폰트, 아이콘 패키지, 과도한 애니메이션 라이브러리를 기본값으로 추가하지 않는다.
- AI 패널은 `보조 도구` 로 보여야지 `대화 앱` 처럼 보이면 안 된다.
- 사용자가 별도로 지시하지 않아도 purple 계열 미래지향 UI, pure white 배경, glass 카드 남발, dark mode 우선 디자인을 만들지 않는다.
- 이미지가 꼭 필요하지 않은 운영형 화면에서 stock photo hero를 넣지 않는다.
- 새 컴포넌트는 데스크톱, `1240px` 이하, `860px` 이하, `720px` 이하 상태를 함께 고려한다.

## 9. 금지 패턴

- 보라색 중심의 AI SaaS 템플릿 룩
- full dark mode를 기본 전제로 한 화면
- 과도한 3D 오브젝트, 자동 반복 애니메이션, 장식용 파티클
- 테이블과 카드에 진한 border를 여러 겹 두르는 방식
- 한 화면에 3개 이상의 강조색 사용
- AI 제안과 사용자 원본 입력을 섞어 보여주는 배치
- 버튼, 탭, 링크가 모두 서로 다른 스타일 언어를 갖는 상태
- 모바일에서 터치 타깃이 너무 작아지는 조밀한 toolbar

## 10. 구현 체크리스트

프론트엔드 화면 작업 전에 아래를 점검한다.

1. 이 화면이 6개 메뉴 중 어디에 속하는지 정했는가
2. hero, panel, card, list, AI panel 중 어떤 패턴 조합을 쓸지 정했는가
3. 액센트 색을 녹색 계열 하나로 유지했는가
4. 본문과 버튼이 최소 가독성 크기를 지키는가
5. hover/focus/active/success/error 상태가 모두 정의돼 있는가
6. AI 관련 요소가 자동 적용처럼 오해되지 않는가
7. `1240px`, `860px`, `720px` 이하에서 정보 우선순위가 유지되는가
8. 그림자, blur, gradient가 내용보다 튀지 않는가
