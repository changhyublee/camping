# 프롬프트 설계

## 1. 문서 목적

이 문서는 로컬 API가 OpenAI Responses API를 통해 캠핑 분석을 수행할 때 사용할 프롬프트 구조를 정의한다.

목표는 다음과 같다.

- 프롬프트를 즉흥적으로 쓰지 않도록 기준을 고정
- 입력 파일과 출력 형식을 일관되게 유지
- 분석 결과 품질을 안정화
- UI와 API가 같은 판단 규칙을 재사용하게 함

## 2. 프롬프트 설계 원칙

### 원칙 1. 역할을 명확히 준다

OpenAI 응답은 단순한 요약기가 아니라 캠핑 준비 플래너 역할을 수행해야 한다.

핵심 역할:

- 준비물 분석가
- 연령대별 개인 준비물 추천자
- 캠핑 식단 제안자
- 이동 동선 및 주변 추천 플래너
- 누락/리스크 점검자

### 원칙 2. 입력과 추론을 분리한다

프롬프트 안에서 아래를 구분해야 한다.

- 사용자 입력 사실
- 문서 기준 규칙
- AI가 추론해야 하는 부분

특히 개인 준비물은 직접 입력값이 아니라 동행자 정보와 조건을 바탕으로 도출해야 한다는 점을 명시해야 한다.

### 원칙 3. 불확실성을 표시한다

날씨, 장소 정보, 외부 데이터가 부족한 경우 추측처럼 단정하지 말고 제한사항을 밝혀야 한다.

### 원칙 4. 출력은 실행 가능해야 한다

결과는 예쁜 설명보다 실제 준비에 도움이 되는 형태여야 한다.

예:

- 챙길 것
- 부족한 것
- 점검할 것
- 추천 메뉴
- 이동 중 들를 곳

## 4. 시스템 프롬프트 초안

아래는 시스템 프롬프트 초안이다.

```md
당신은 캠핑 준비를 분석하는 AI 플래너다.

당신의 목표는 사용자가 더 즐거운 캠핑을 할 수 있도록
캠핑 장비, 연령대별 개인 준비물, 식단/요리, 이동 동선,
캠핑장 주변 즐길거리와 맛집, 출발 전 체크리스트를 종합적으로 추천하는 것이다.

분석 시 반드시 지켜야 할 규칙:

1. 개인 준비물은 사용자가 직접 입력한 목록으로 간주하지 말고,
   동행자 연령대, 건강 특이사항, 날씨, 기온, 장소 조건을 바탕으로 추천하라.
2. 장비 추천은 보유 장비, 소모품, 사전 점검 상태를 함께 고려하라.
3. 요리 추천은 인원, 날씨, 조리 가능 조건, 음식 취향을 함께 고려하라.
4. 이동/주변 추천은 일정 부담, 동행자 특성, 날씨를 함께 고려하라.
5. 정보가 부족하면 추정이라고 밝히고 제한사항을 명확히 표시하라.
6. 결과는 사용자가 바로 준비에 활용할 수 있도록 체크리스트 형태를 포함하라.
```

## 5. 분석 규칙 초안

분석 규칙은 시스템 프롬프트와 별도로 유지하는 것을 권장한다.

예시:

```md
분석 규칙:

- 날짜와 장소가 모두 있으면 가장 구체적인 추천을 수행한다.
- 날짜만 있으면 계절성과 기온 중심의 범용 추천을 수행한다.
- 장소만 있으면 장소 특성 중심 추천을 수행하고 계절 의존 항목은 조건부로 제안한다.
- 어린 동행자가 있으면 보온, 여벌 옷, 비상약, 간식, 휴식 요소를 더 보수적으로 추천한다.
- 밤 기온이 낮으면 보온 장비와 따뜻한 식단 비중을 높인다.
- 우천 가능성이 있으면 우비, 방수 수납, 여벌 옷, 대체 실내 코스를 우선 제안한다.
- 소모품 부족 상태면 추천 준비물에 경고를 포함한다.
- 차량 적재 여유가 적으면 우선순위가 낮은 장비를 제외 후보로 제안한다.
```

## 6. 입력 데이터 묶음 구성

분석 시 입력되는 문맥은 아래 묶음으로 구성한다.

### 문서 규칙

- `README.md`
- `docs/requirements.md`
- `docs/technical-architecture.md`
- `docs/data-model.md`
- `docs/trip-analysis-workflow.md`

### 프롬프트 소스

- `prompts/system.md`
- `prompts/trip-analysis.md`
- 필요 시 `prompts/output-format.md`

### 사용자 공통 데이터

- `profile.yaml`
- `companions.yaml`
- `equipment/*.yaml`
- `preferences/*.yaml`

### 캠핑 요청 데이터

- `trips/<trip-id>.yaml`

### 선택적 외부 데이터

- 날씨 캐시
- 장소 캐시

## 7. 출력 형식 규칙

결과는 아래 섹션을 기본으로 권장한다.

```md
# 캠핑 분석 결과

## 1. 요약
## 2. 추천 장비
## 3. 연령대별 개인 준비물
## 4. 부족한 장비/소모품
## 5. 출발 전 체크리스트
## 6. 추천 식단/요리
## 7. 이동 중 추천 장소
## 8. 캠핑장 주변 추천
## 9. 주의사항 및 한계
```

## 8. 출력 스타일 규칙

- 항목은 짧고 실용적으로 작성
- 추천 이유를 함께 제공
- 누락 시 발생할 위험을 설명
- 연령대별 항목은 묶어서 보여줌
- 불확실성은 별도 섹션 또는 문장으로 표시

## 9. 프롬프트 로딩 전략

v1에서는 프롬프트를 최소 2개로 나누는 것을 권장한다.

### `prompts/system.md`

역할:

- 프로젝트 정체성
- 분석 원칙
- 판단 기준

### `prompts/trip-analysis.md`

역할:

- 특정 trip 분석 지시
- 출력 포맷 지시
- 누락 정보 처리 규칙

추후 아래를 추가할 수 있다.

- `prompts/output-format.md`
- `prompts/retry-refinement.md`

## 10. 로컬 API 조합 방식

로컬 API는 아래 순서로 프롬프트를 조합하는 것을 권장한다.

1. 시스템 프롬프트 로드
2. trip 분석 프롬프트 로드
3. 관련 문서 규칙 요약 결합
4. 사용자 데이터와 trip 데이터 직렬화
5. 저장 여부, 재분석 지시, 특정 강조 조건을 추가

원칙:

- 프롬프트 파일을 코드에 하드코딩하지 않는다
- 프롬프트 수정만으로 분석 방향을 보정할 수 있어야 한다

## 11. 예시 요청 문맥

로컬 API는 내부적으로 아래 성격의 요청을 만들게 된다.

```text
시스템 프롬프트와 분석 규칙을 따르고,
trip 요청 파일과 companions/equipment/preferences 데이터를 바탕으로
이번 캠핑에 대한 장비, 연령대별 개인 준비물, 요리, 이동 중 추천 장소,
캠핑장 주변 추천, 체크리스트를 Markdown으로 정리해라.
부족한 정보는 한계로 표시하고 추정은 추정으로 밝힌다.
```

## 12. 안티 패턴

아래 방식은 피해야 한다.

- 개인 준비물을 사용자 직접 입력 목록처럼 다루는 프롬프트
- 장비 추천만 하고 식단/이동/주변 추천을 빼는 프롬프트
- 정보 부족 상태를 숨기고 단정적으로 추천하는 프롬프트
- 연령대 구분 없이 모든 동행자에게 같은 개인 준비물을 제안하는 프롬프트
- 브라우저 코드에 프롬프트 문자열을 중복 보관하는 구조

## 13. v1 권장 프롬프트 목표

v1 프롬프트는 아래 수준만 만족하면 된다.

- 준비물 추천
- 연령대별 개인 준비물 추천
- 메뉴 추천
- 이동 및 주변 추천
- 누락 정보와 리스크 표시

즉, 고급 모델 튜닝보다 `명확한 입력 구조와 출력 형식` 이 우선이다.

## 14. 현재 결정 사항

- 프롬프트는 역할, 규칙, 입력, 출력 형식으로 분리한다
- 개인 준비물은 조건 기반 추천 결과로 명시한다
- 정보 부족 시 제한사항을 출력에 포함한다
- 프롬프트는 `prompts/` 디렉토리로 분리한다
- 로컬 API가 프롬프트 파일을 읽어 OpenAI 요청을 구성한다
