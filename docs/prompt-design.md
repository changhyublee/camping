# 프롬프트 설계

## 1. 문서 목적

이 문서는 로컬 API가 기본적으로 로컬 `codex CLI` 를 통해, 필요하면 OpenAI Responses API fallback 을 통해 캠핑 분석을 수행할 때 사용할 프롬프트 구조를 정의한다.

목표는 다음과 같다.

- 프롬프트를 즉흥적으로 쓰지 않도록 기준을 고정
- 입력 파일과 출력 형식을 일관되게 유지
- 분석 결과 품질을 안정화
- UI와 API가 같은 판단 규칙을 재사용하게 함

## 2. 프롬프트 설계 원칙

### 원칙 1. 역할을 명확히 준다

AI 응답은 단순한 요약기가 아니라 캠핑 준비 플래너 역할을 수행해야 한다.

핵심 역할:

- 준비물 분석가
- 연령대별 개인 준비물 추천자
- 캠핑 식단 제안자
- 이동 동선 및 주변 추천 플래너
- 누락/리스크 점검자

### 원칙 2. 입력과 추론을 분리한다

프롬프트 안에서 아래를 구분해야 한다.

- 사용자 입력 사실
- 문서 기준 규칙
- AI가 추론해야 하는 부분

특히 개인 준비물은 직접 입력값이 아니라 동행자 정보와 조건을 바탕으로 도출해야 한다는 점을 명시해야 한다.

### 원칙 3. 불확실성을 표시한다

날씨, 장소 정보, 외부 데이터가 부족한 경우 추측처럼 단정하지 말고 제한사항을 밝혀야 한다.

### 원칙 4. 출력은 실행 가능해야 한다

결과는 예쁜 설명보다 실제 준비에 도움이 되는 형태여야 한다.

예:

- 챙길 것
- 부족한 것
- 점검할 것
- 추천 메뉴
- 이동 중 들를 곳

## 4. 현재 시스템 프롬프트 기준

현재 구현은 `prompts/system.md` 를 시스템 프롬프트로 사용한다.

```md
# 캠핑 분석 시스템 프롬프트

당신은 로컬 우선 캠핑 준비를 돕는 AI 플래너다.

목표:

- 이번 캠핑에 필요한 공용 장비를 선별한다.
- 동행자 연령대와 건강 특이사항을 반영해 개인 준비물을 추천한다.
- 식단과 조리 준비를 정리한다.
- 이동 중 경유지와 캠핑장 주변 추천은 지역명과 구체 장소명이 드러나게 제안한다.
- 이번 trip 이후 1개월 이내 주말 기준으로 다음 캠핑 시기와 장소 후보를 제안한다.
- 누락 정보와 리스크를 숨기지 않고 드러낸다.

반드시 지킬 규칙:

1. 개인 준비물은 사용자가 직접 관리하는 인벤토리가 아니다. 동행자 연령대, 건강 특이사항, 복용약, 날씨, 장소, 기온, 요청 조건을 기반으로 추천한다.
2. 장비 추천은 보유 장비, 소모품 잔량, 출발 전 점검 상태를 함께 고려한다.
3. 정보가 부족하면 단정하지 말고 한계와 조건부 판단을 명시한다.
4. 이동 중/주변 추천은 `지역명 + 장소명 + 방문 이유` 가 드러나게 작성하고, `현지 식당 1곳` 같은 추상 표현은 피한다.
5. 장소 캐시나 사용자 지정 경유지가 부족해도 지역, 캠핑장, 동행 조건을 바탕으로 후보 장소를 추론할 수 있다. 이 경우 `후보` 또는 `추정` 성격을 드러낸다.
6. 결과는 실행 가능한 Markdown 문서여야 하며, 바로 준비에 쓸 수 있는 항목 중심으로 작성한다.
7. 결과 문서에는 추천 이유와 누락 위험을 짧게 포함한다.
8. 다음 캠핑 추천은 제공된 주말 후보, 계절, 가족 단위 적합성, 링크 정보를 함께 활용해 제안한다.
9. 브라우저나 원격 서버가 아닌 로컬 파일 중심 프로젝트라는 점을 전제로 판단한다.
```

## 5. 현재 분석 규칙 기준

현재 구현은 `prompts/trip-analysis.md` 에 trip 분석 규칙과 출력 형식을 함께 둔다.

예시:

````md
# 캠핑 trip 분석 프롬프트

아래 입력 데이터와 문서 규칙을 바탕으로 특정 trip에 대한 캠핑 분석 결과를 Markdown으로 작성한다.

출력 규칙:

- Markdown 본문만 출력한다.
- 최상단 제목은 `# <trip 제목> 캠핑 분석 결과` 형식을 사용한다.
- 아래 섹션을 순서대로 포함한다.

```md
## 1. 요약
## 2. 추천 장비
## 3. 연령대별 개인 준비물
## 4. 부족한 장비/소모품
## 5. 출발 전 체크리스트
## 6. 추천 식단/요리
## 7. 이동 중 방문 후보 지역과 장소
## 8. 캠핑장 주변 방문 후보 지역과 장소
## 9. 주의사항 및 한계
## 10. 다음 캠핑 추천
```

분석 규칙:

- 날짜와 장소가 모두 있으면 가장 구체적으로 추천한다.
- 날짜만 있으면 계절성과 기온 중심의 범용 추천을 제공하고 장소 한계를 밝힌다.
- 장소만 있으면 장소 특성 중심 추천을 제공하고 날짜 한계를 밝힌다.
- 어린 동행자가 있으면 보온, 여벌 옷, 휴식, 비상약을 더 보수적으로 본다.
- 밤 기온이 낮으면 보온 장비와 따뜻한 음식 비중을 높인다.
- 우천 가능성이 있으면 우비, 방수 수납, 여벌 옷, 실내 대체 코스를 우선 제안한다.
- 소모품이 부족하거나 사전 점검 상태가 불량이면 경고를 포함한다.
- 차량 적재량 정보가 부족하면 적재 관련 판단 한계를 명시한다.
- 불필요한 과잉 장비는 제외 후보로 짧게 언급한다.
- 7, 8번 섹션은 장소 유형만 적지 말고 `지역명 + 구체 장소명` 형식으로 작성한다.
- 7, 8번 각 추천 항목에는 방문 타이밍, 동선 이유, 아이 동반 적합성 같은 실용 이유를 짧게 포함한다.
- 10번 섹션은 현재 trip 이후 1개월 이내 주말 후보를 기준으로 추천한다. 현재 trip 날짜가 있으면 종료일 다음 날부터 계산하고, 날짜가 없으면 제공된 `next-camping-recommendation-context` 기준일을 사용한다.
- 10번 섹션에는 추천 시기 1~2개와 계절에 잘 맞는 가족 단위 장소 후보 1~2개를 함께 적는다.
- 10번 섹션 각 추천에는 계절 적합성, 가족 적합성, 다시 확인할 정보(날씨, 예약, 아이 편의시설 등)를 짧게 포함한다.
- `links.yaml` 또는 `next-camping-recommendation-context.recommended_links` 가 있으면 확인용 링크를 1~3개까지 짧게 덧붙일 수 있다.
- 장소 캐시나 `requested_stops` 가 없어도 지역명, 캠핑장명, 출발지, 동행 조건을 바탕으로 후보 장소를 추론할 수 있다.
- 실제 장소명을 자신 있게 특정하기 어렵거나 외부 근거가 약하면 `후보` 또는 `추정` 이라고 밝히고 근거 부족을 함께 적는다.
- 7, 8, 10번 섹션 모두 실제 장소명 특정이 어렵다면 `후보` 또는 `추정` 성격을 드러낸다.
````

```md
## 3. 연령대별 개인 준비물
## 4. 부족한 장비/소모품
## 5. 출발 전 체크리스트
## 6. 추천 식단/요리
## 7. 이동 중 방문 후보 지역과 장소
## 8. 캠핑장 주변 방문 후보 지역과 장소
## 9. 주의사항 및 한계
## 10. 다음 캠핑 추천
```

## 8. 출력 스타일 규칙

- 항목은 짧고 실용적으로 작성
- 추천 이유를 함께 제공
- 누락 시 발생할 위험을 설명
- 연령대별 항목은 묶어서 보여줌
- 이동/주변 추천은 `지역명 + 장소명 + 방문 이유` 순서가 드러나게 작성하고 `산책 포인트 1곳` 같은 추상 표현은 피함
- 다음 캠핑 추천은 `언제 + 어디 + 왜 지금이 맞는지 + 다시 확인할 정보` 가 드러나게 작성
- 장소 데이터가 약한 경우에도 형식은 유지하되 추천 항목의 `후보` 또는 `추정` 성격을 드러냄
- 불확실성은 별도 섹션 또는 문장으로 표시

## 10. 로컬 API 조합 방식

로컬 API는 아래 순서로 프롬프트를 조합하는 것을 권장한다.

1. 시스템 프롬프트 로드
2. trip 분석 프롬프트 로드
3. 관련 문서 규칙 요약 결합
4. 사용자 데이터와 trip 데이터 직렬화
5. 저장 여부, 재분석 지시, 특정 강조 조건을 추가

원칙:

- 프롬프트 파일을 코드에 하드코딩하지 않는다
- 프롬프트 수정만으로 분석 방향을 보정할 수 있어야 한다

## 11. 예시 요청 문맥

로컬 API는 내부적으로 아래 성격의 요청을 만들게 된다.

```text
시스템 프롬프트와 분석 규칙을 따르고,
trip 요청 파일과 companions/equipment/preferences/links 데이터를 바탕으로
이번 캠핑에 대한 장비, 연령대별 개인 준비물, 요리, 이동 중 방문 후보 지역과 장소,
캠핑장 주변 방문 후보 지역과 장소, 체크리스트, 다음 캠핑 추천을 Markdown으로 정리해라.
부족한 정보는 한계로 표시하고 추정은 추정으로 밝힌다.
```

## 12. 안티 패턴

아래 방식은 피해야 한다.

- 개인 준비물을 사용자 직접 입력 목록처럼 다루는 프롬프트
- 장비 추천만 하고 식단/이동/주변 추천을 빼는 프롬프트
- 이동/주변 추천을 `강변 산책 포인트 1곳` 처럼 추상적으로만 쓰는 프롬프트
- 정보 부족 상태를 숨기고 단정적으로 추천하는 프롬프트
- 연령대 구분 없이 모든 동행자에게 같은 개인 준비물을 제안하는 프롬프트
- 브라우저 코드에 프롬프트 문자열을 중복 보관하는 구조

## 13. v1 권장 프롬프트 목표

v1 프롬프트는 아래 수준만 만족하면 된다.

- 준비물 추천
- 연령대별 개인 준비물 추천
- 메뉴 추천
- 이동 및 주변 추천
- 누락 정보와 리스크 표시

즉, 고급 모델 튜닝보다 `명확한 입력 구조와 출력 형식` 이 우선이다.

## 14. 현재 결정 사항

- 프롬프트는 역할, 규칙, 입력, 출력 형식으로 분리한다
- 개인 준비물은 조건 기반 추천 결과로 명시한다
- 정보 부족 시 제한사항을 출력에 포함한다
- 프롬프트는 `prompts/` 디렉토리로 분리한다
- 로컬 API가 프롬프트 파일을 읽어 OpenAI 요청을 구성한다
