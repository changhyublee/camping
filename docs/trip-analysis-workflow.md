# 캠핑 분석 워크플로우

## 1. 문서 목적

이 문서는 사용자가 UI에서 계획을 만들고, 장비를 점검하고, AI 보조와 분석을 거쳐, 최종적으로 히스토리로 아카이브하는 전체 흐름을 정의한다.

## 2. 핵심 원칙

- 분석의 기준 파일은 `trips/*.yaml`
- 장비와 링크는 계획과 별도 파일로 관리한다
- AI 보조는 제안만 하고 자동 저장하지 않는다
- 분석 결과는 `outputs/*.md` 로 저장한다
- 분석 상태는 `cache/analysis-jobs/*.json` 으로 저장한다
- 섹션별 누적 결과는 `cache/analysis-results/*.json` 으로 저장한다
- 완료된 계획은 `history/*.yaml` 로 이동한다
- 히스토리 회고 원문은 `history/*.yaml` 의 `retrospectives` 배열에 누적 저장한다
- 히스토리별 AI 학습 결과는 `cache/history-learning/*.json` 으로 저장한다
- 모든 회고를 합산한 전역 개인화 학습 프로필은 `cache/user-learning/profile.json` 으로 저장하고 이후 분석/보조에 자동 반영한다

## 3. 기본 실행 흐름

### 1단계. 기본 데이터 관리

사용자는 먼저 아래 데이터를 관리한다.

- `companions.yaml`
- `vehicles.yaml`
- `equipment/*.yaml`
- `preferences/*.yaml`
- `links.yaml`

현재 UI에서는 `사람 관리`, `차량 관리`, `장비 관리`, `외부 링크` 화면에서 기준 데이터를 먼저 정리한다.

### 2단계. 캠핑 계획 작성

사용자는 `캠핑 계획` 화면에서 아래를 입력한다.

- 날짜
- 장소
- 동행자 선택
- 차량 선택
- 날씨와 취사/전기 조건
- 식단/이동 요청

동행자와 차량은 계획 화면에서 직접 새로 만들지 않는다. 기준 데이터가 비어 있으면 각각 `사람 관리`, `차량 관리` 화면에서 먼저 등록한 뒤 계획에 선택한다.

### 3단계. AI 보조 사용

사용자는 저장된 계획을 기준으로 AI 보조에 질문한다.

AI 보조는:

- 지금 부족한 정보 경고
- 장비 보강 제안
- 점검 항목 제안
- 폼에서 먼저 바꿀 값 안내

를 반환한다.

### 4단계. 분석 실행

사용자는 `분석 섹션` 에서 필요한 항목만 체크해 `선택 수집` 또는 `전체 실행` 을 누른다.

로컬 API는:

1. 계획 파일을 읽는다
2. 공통 데이터와 장비 파일을 읽는다
3. 요청된 섹션을 개별 job 으로 queue 에 넣고 상태 파일에 반영한다
4. 같은 `trip_id` 에서는 한 번에 하나의 섹션만 `running` 으로 실행하고 나머지는 `queued` 로 둔다
5. 백그라운드에서 검증 경고를 생성하고, 요청된 섹션 범위만 포함하는 프롬프트를 만든다
6. 로컬 `codex exec` 또는 fallback AI 백엔드를 호출한다
7. 섹션별 fragment Markdown을 `cache/analysis-results/<trip-id>.json` 에 누적 저장한다
8. 섹션 하나가 끝날 때마다 최종 결과를 다시 합성해 `outputs/<trip-id>-plan.md` 를 덮어쓴다
9. 실행 직전 `running`, queue 가 남아 있으면 `queued`, 모든 요청이 끝나면 `completed` 또는 실패 시 `failed` 로 상태를 갱신한다

UI 동작 원칙:

- 섹션마다 `미수집`, `대기 중`, `수집 중`, `수집 완료`, `실패`, `중단` 칩을 보여 준다
- `queued` 또는 `running` 이면 상단 주요 버튼 라벨을 `분석 중...` 으로 유지한다
- 이미 `수집 완료` 인 섹션은 그대로 표시하고, 필요할 때만 `재수집` 을 눌러 다시 queue 에 넣는다
- `원본 입력` 탭 제목 위 `날씨 자동 수집` 카드의 `수집` 버튼은 지역과 일정으로 Google 검색을 실행하고 빠른 보조 웹 조사 AI 모델이 날씨 입력란을 채운다
- 날씨를 비워 둔 채 계획을 저장하면 서버가 저장 직후 백그라운드 날씨 수집을 시작하고, 완료되면 `conditions.expected_weather` 와 `cache/weather/<trip-id>-weather.json` 을 갱신한다
- 계획의 동행자 summary 카드에서 메일 주소가 있는 사람만 발송 대상으로 체크할 수 있다
- pending 분석, 메타데이터 수집, 개인화 학습 작업이 있을 때만 SSE 실시간 이벤트로 진행 상태를 즉시 갱신하고, 모두 idle 이면 연결을 닫는다
- 새로고침 후 다시 들어오거나 재연결되면 상태 조회 API로 현재 상태를 복원한다
- 좌측 관리 설정 영역의 `모든 AI 요청 중단` 버튼은 실행 중인 분석과 장비 메타데이터 수집, 개인화 학습을 함께 중단하고 남아 있던 queue 를 비운다

### 5단계. 결과 저장

분석 결과는 섹션 하나가 끝날 때마다 자동으로 다시 합성된다.

같은 `trip_id` 의 기존 결과가 있으면 같은 경로에 자동으로 덮어쓴다.

분석 도중에는 기존 결과 Markdown을 지우지 않는다. UI는 마지막으로 저장된 결과를 계속 보여 주고, 완료되면 같은 경로의 새 파일을 다시 읽는다.

저장 위치:

- `.camping-data/outputs/<trip-id>-plan.md`

상태 저장 위치:

- `.camping-data/cache/analysis-jobs/<trip-id>.json`

섹션별 누적 결과 저장 위치:

- `.camping-data/cache/analysis-results/<trip-id>.json`

재시작 복구 원칙:

- API 서버가 재시작되면 남아 있던 `queued` 또는 `running` 상태는 `interrupted` 로 전환한다
- `interrupted` 상태는 자동 재개하지 않고 사용자가 다시 `선택 수집`, `전체 실행`, `재수집` 중 하나를 눌러 새 작업을 시작한다

### 5-1단계. 분석 결과 메일 발송

전체 분석 항목이 모두 모인 뒤 사용자는 `원본 입력` 탭에서 메일을 보낼 동행자만 체크한다.

발송 버튼은 `원본 입력` 탭과 `AI·결과 > 분석 결과` 탭 두 곳에서 모두 제공한다.

이때:

- 체크하지 않은 동행자에게는 메일을 보내지 않는다
- 메일 주소가 없는 동행자는 체크할 수 없다
- 복수 선택 시에도 한 번의 발송 요청으로 함께 처리한다
- 현재 선택 상태는 `trips/<trip-id>.yaml` 의 `notifications.email_recipient_companion_ids` 로 저장한다
- 메일 본문은 저장된 `outputs/<trip-id>-plan.md` 전체 Markdown을 그대로 사용한다

### 6단계. 히스토리 아카이브

캠핑이 끝난 뒤 사용자는 계획을 히스토리로 이동한다.

분석이 `queued` 또는 `running` 상태이면 삭제와 히스토리 이동은 허용하지 않는다.

이때:

- `trips/<trip-id>.yaml` 는 계획 목록에서 제거된다
- `history/<trip-id>.yaml` 가 생성된다
- 결과 문서 경로는 히스토리에 참조로 남는다
- 당시 선택한 동행자와 차량 요약은 히스토리 스냅샷으로 함께 남는다

### 7단계. 히스토리 회고와 개인화 학습

캠핑을 다녀온 뒤 사용자는 히스토리 상세에 실제 사용 경험을 추가로 남긴다.

이때 사용자는 아래 정보를 append-only 로 저장할 수 있다.

- 만족도
- 실제로 사용한 반복 장비
- 잘 안 쓴 것과 과했던 준비물
- 부족했던 장비나 소모품
- 식단/요리 회고
- 이동/동선 회고
- 캠핑장/사이트 회고
- 문제/이슈
- 다음에 더 하고 싶은 점
- 자유 후기

로컬 API는:

1. `history/<history-id>.yaml` 의 `retrospectives` 배열에 회고 엔트리를 추가한다
2. 해당 히스토리의 AI 회고 분석 결과를 `cache/history-learning/<history-id>.json` 으로 갱신한다
3. 모든 히스토리 학습 결과를 바탕으로 전역 개인화 학습 프로필을 `cache/user-learning/profile.json` 에 다시 합성한다
4. 전역 개인화 학습 상태를 `cache/user-learning/jobs/profile.json` 에 저장한다
5. `GET /api/ai-jobs/events` 의 `user-learning-status` 이벤트로 UI를 즉시 갱신한다

UI 동작 원칙:

- 회고 원문 저장은 항상 먼저 성공해야 한다
- 개인화 학습이 실패해도 원문 회고와 마지막 성공한 학습 프로필은 유지한다
- 히스토리 상세는 `이번 캠핑에서 AI가 배운 점` 과 `전역 개인화 학습 요약` 을 함께 보여 준다
- 이후 계획 분석과 AI 보조는 전역 개인화 학습 프로필을 자동으로 함께 읽는다
