# 캠핑 분석 워크플로우

## 1. 문서 목적

이 문서는 로컬 파일 기반 구조에서 사용자가 UI로 캠핑 정보를 입력하거나 선택하고, 로컬 API가 이를 읽어 기본적으로 로컬 `codex exec` 응답을 요청한 뒤, 결과를 출력하고 저장하는 전체 흐름을 정의한다.

대상 범위:

- 원격 서버 없이 동작하는 v1 워크플로우
- `./.camping-data/` 기준 로컬 파일 입력/출력
- 로컬 웹 UI와 로컬 API 조합
- 캠핑 준비물, 연령대별 개인 준비물, 요리, 이동 동선, 주변 추천 분석 흐름

## 2. 전제 조건

이 워크플로우는 아래 전제를 따른다.

- 사용자 데이터는 `./.camping-data/` 에 저장한다
- 사용자가 직접 입력하는 파일은 YAML이다
- 분석 결과는 Markdown으로 저장한다
- GitHub는 운영 저장소가 아니라 코드/문서 저장소다
- 브라우저 UI는 로컬 API를 통해서만 분석을 실행한다
- 로컬 API가 기본적으로 로컬 `codex exec` 호출을 담당한다
- 필요하면 OpenAI Responses API를 fallback 으로 사용할 수 있다

## 3. 워크플로우의 핵심 원칙

### 원칙 1. 분석은 캠핑 요청 파일 단위로 실행한다

실제 분석의 시작점은 `trips/*.yaml` 파일이다.

이 파일에는 최소한 아래 정보가 들어간다.

- 날짜
- 장소
- 동행자
- 차량
- 조건
- 식사/이동 관련 요청

### 원칙 2. 개인 준비물은 입력값이 아니라 분석 결과다

사용자는 `개인 준비물 목록`을 직접 작성하지 않는다.

대신 아래를 입력한다.

- 동행자의 연령대
- 건강 특이사항
- 복용약
- 날씨/온도 조건
- 장소 조건

그리고 AI 응답이 이 정보를 바탕으로 연령대별 개인 준비물을 추천한다.

### 원칙 3. 외부 정보는 선택적으로 결합한다

v1에서는 아래 두 방식 모두 허용한다.

- 사용자가 날씨/장소 정보를 수동으로 입력
- 추후 스크립트나 수집 로직이 캐시에 저장한 외부 정보를 읽어 사용

즉, 외부 API가 없어도 분석은 가능해야 한다.

### 원칙 4. UI는 표시와 입력을 담당하고 로컬 API는 규칙 집행을 담당한다

- UI는 사용자의 입력, 실행 상태, 결과 표시를 담당한다
- 로컬 API는 검증, 컨텍스트 조합, AI 백엔드 호출, 저장을 담당한다

## 4. 주요 역할

### 사용자

- 로컬 UI에서 trip을 선택하거나 입력한다
- 분석 실행을 요청한다
- 결과를 검토하고 필요하면 입력값을 수정해 재실행한다

### 로컬 웹 UI

- trip 목록과 상세 정보를 보여준다
- 분석 요청을 로컬 API에 전달한다
- 진행 상태, 오류, 경고를 표시한다
- Markdown 결과를 화면에 렌더링한다

### 로컬 API

- 관련 문서와 로컬 데이터를 읽는다
- 입력값의 누락과 모순을 점검한다
- AI 백엔드 요청용 컨텍스트를 생성한다
- 결과를 UI에 반환하고 파일로 저장한다

### 로컬 codex CLI 또는 fallback AI 백엔드

- 캠핑 준비물, 연령대별 개인 준비물, 요리, 이동/주변 추천을 생성한다
- 정보 부족 시 제한사항과 리스크를 함께 표현한다

### 보조 스크립트

추후 추가될 수 있는 역할:

- YAML 구조 검증
- 입력 파일 템플릿 생성
- 캐시 파일 로드
- 결과 저장 자동화

## 6. 기본 실행 흐름

### 6.1 1단계: 사용자 기본 데이터 준비

사용자는 먼저 자주 바뀌지 않는 공통 데이터를 준비한다.

대상 파일:

- `profile.yaml`
- `companions.yaml`
- `equipment/*.yaml`
- `preferences/*.yaml`

이 단계의 목적:

- 매번 반복 입력하지 않도록 공통 정보를 축적
- 장비, 동행자, 취향을 재사용 가능하게 구성

### 6.2 2단계: 캠핑 요청 파일 작성 또는 선택

사용자는 특정 캠핑 일정에 대한 요청 파일을 UI에서 선택하거나 작성한다.

예:

```text
.camping-data/trips/2026-04-18-gapyeong.yaml
```

이 파일에는 최소한 아래를 넣는다.

- 날짜 또는 장소 중 하나
- 참여 동행자 ID

아래 항목은 있으면 분석 품질이 좋아지지만 없어도 v1에서는 조건부 분석이 가능하다.

- 출발지
- 사용할 차량
- 전기/취사 가능 여부
- 식사 계획 또는 AI 추천 여부
- 이동 계획 또는 AI 추천 여부

### 6.3 3단계: 분석 요청 실행

사용자는 UI에서 특정 trip 파일을 기준으로 분석을 요청한다.

로컬 UI는 아래 정보를 로컬 API에 전달한다.

- 대상 `trip_id`
- 선택적 추가 지시
- 저장 여부

로컬 API는 `POST /api/analyze-trip` 형태로 요청을 받는 것을 기본으로 한다.

### 6.4 4단계: 입력 검증

로컬 API는 분석 전에 아래를 점검해야 한다.

- `trip_id` 가 데이터 모델 규칙에 맞는가
- trip 파일명과 내부 `trip_id` 가 일치하는가
- trip 파일에 날짜가 있는가
- 장소 정보가 있는가
- companion ID가 `companions.yaml` 에 존재하는가
- 차량 정보가 충분한가
- 취사 가능 여부가 식단 계획과 충돌하지 않는가
- 필요한 장비/소모품 파일이 존재하는가

검증 결과에 따라:

- 치명적 누락: `날짜와 장소가 모두 없음`, `동행자 정보 없음`, `잘못된 trip_id` 같은 경우 분석 중단 후 UI에 보완 요청
- 일부 누락: 제한사항을 명시하고 분석 계속

### 6.5 5단계: 컨텍스트 조합

로컬 API는 분석용 컨텍스트를 아래 순서로 조합한다.

1. 프로젝트 목적과 판단 기준
2. 데이터 모델 규칙
3. 프롬프트 규칙
4. 사용자 공통 데이터
5. 특정 trip 요청
6. 선택적 외부 캐시 데이터

권장 읽기 우선순위:

1. `README.md`
2. `docs/requirements.md`
3. `docs/technical-architecture.md`
4. `docs/data-model.md`
5. `prompts/system.md`
6. `prompts/trip-analysis.md`
7. `.camping-data/trips/<trip-id>.yaml`
8. 나머지 로컬 사용자 데이터
9. 필요한 경우 캐시 파일

### 6.6 6단계: 로컬 codex CLI 또는 fallback AI 백엔드 분석 수행

로컬 API는 아래 항목을 묶어서 기본적으로 `codex exec` 요청을 구성하고, 필요하면 fallback 으로 OpenAI Responses API를 사용할 수 있다.

#### A. 캠핑 장비 분석

- 가져가야 할 장비
- 불필요한 장비
- 부족한 장비
- 부족한 소모품
- 출발 전 점검 항목

#### B. 연령대별 개인 준비물 분석

- 동행자의 연령대별 필요 준비물
- 날씨/온도/장소 조건 반영
- 건강 특이사항과 복용약 반영

#### C. 요리/식단 분석

- 사용자가 입력한 메뉴 검토
- 날씨와 인원에 맞는 보완 메뉴 추천
- 필요한 식재료와 조리도구 정리

#### D. 여정 분석

- 이동 중 들를 만한 장소
- 이동 중 식사 장소
- 캠핑장 주변 즐길거리
- 캠핑장 주변 맛집/카페/마트
- 우천 시 대체 코스

#### E. 종합 위험/체크 분석

- 밤 기온 대비 부족 여부
- 우천 대비 부족 여부
- 차량 적재량 부담
- 동행자 조건 대비 준비 부족

### 6.7 7단계: 결과 반환과 저장

로컬 API는 분석 결과를 Markdown으로 정리하고 UI에 반환한다.

권장 결과 구조:

1. 캠핑 개요
2. 핵심 판단 요약
3. 추천 장비 목록
4. 연령대별 개인 준비물 목록
5. 부족한 물품 목록
6. 출발 전 체크리스트
7. 추천 식단/요리
8. 이동 중 추천 장소
9. 캠핑장 주변 추천 장소
10. 주의사항 및 리스크

저장 경로 예:

```text
.camping-data/outputs/2026-04-18-gapyeong-plan.md
```

저장 시점:

- 사용자가 저장 옵션을 켠 채 실행한 경우
- 또는 결과 확인 후 별도 저장 버튼을 누른 경우

### 6.8 8단계: 사용자 검토 및 재실행

사용자는 결과를 본 뒤 아래 중 하나를 한다.

- 그대로 사용
- 입력값 수정 후 재분석
- 특정 항목만 다시 요청

예:

- 메뉴만 다시 추천
- 개인 준비물만 다시 정교화
- 비 오는 경우 플랜만 다시 생성

이 재실행이 쉬워야 하므로 UI와 로컬 파일 기반 구조가 중요하다.

## 7. 입력 시나리오별 동작 규칙

### 7.1 날짜만 지정한 경우

가능한 동작:

- 계절성 판단
- 예상 기온 범위 판단
- 일반적 준비 방향 제안

제한:

- 장소 특화 추천은 약해진다
- 주변 맛집/관광지 추천 정확도는 떨어진다

처리 원칙:

- 장소 미지정 상태임을 UI와 결과 문서에 명확히 표시
- 범용 추천 중심으로 응답

### 7.2 장소만 지정한 경우

가능한 동작:

- 장소 특성 기반 장비 제안
- 이동 경로와 주변 추천 일부 수행

제한:

- 날씨와 기온 기반 판단은 제한된다
- 계절성이 불명확하면 보수적 추천이 필요하다

처리 원칙:

- 날짜 미지정 상태를 명시
- 계절/날씨 민감 항목은 조건부 표현 사용

### 7.3 날짜와 장소를 함께 지정한 경우

가장 완전한 분석이 가능하다.

우선 분석 대상:

- 날씨/기온 대응
- 장소 특성
- 이동 동선
- 연령대별 준비물
- 식단과 주변 추천

## 8. 누락 정보 처리 규칙

### 필수 누락

아래 항목이 없으면 분석을 중단하거나 보완을 요청한다.

- `trip_id` 형식 오류 또는 경로 안전 규칙 위반
- trip 날짜와 장소 정보가 모두 없음
- 동행자 정보

### 보완 가능 누락

아래 항목은 없어도 분석 가능하지만 제한을 표시해야 한다.

- 날짜 없음
- 장소 없음
- 예상 날씨
- 차량 적재량
- 메뉴 계획
- 외부 캐시 데이터

표현 원칙:

- 무엇이 없어서 어떤 정확도 제한이 생기는지 명시한다
- UI에는 경고 메시지로, 결과 문서에는 한계 섹션으로 보여준다

## 9. 출력 형식 규칙

결과 문서는 사람이 바로 읽고 준비에 사용할 수 있어야 한다.

권장 스타일:

- 짧은 요약 먼저
- 실제 챙길 항목 중심
- 연령대별로 구분
- 체크리스트 형태 포함
- 불확실성 명시

반드시 포함할 문장 성격:

- 추천 이유
- 조건 의존성
- 누락 위험

예:

- 밤 최저기온이 낮아 유치원생용 방한 외투를 추가 추천
- 우천 가능성이 있어 여벌 옷과 우비를 추가 추천
- 현재 보유 장비 기준으로 난방 계열 장비가 부족함

## 10. v1에서 외부 정보 처리 방식

v1에서는 외부 정보 수집을 자동화하지 않아도 된다.

허용되는 방식:

- 사용자가 trip 파일에 예상 날씨를 수동 입력
- 사용자가 별도 메모로 장소 특성을 입력
- 추후 수집된 캐시 파일이 있으면 선택적으로 참조

즉, 분석 품질은 높일 수 있지만 외부 정보가 없어도 동작은 가능해야 한다.

## 11. 예시 워크플로우

### 예시 1. 가족 봄 캠핑

1. 사용자가 `companions.yaml` 에 성인 2명, 유치원생 1명을 등록
2. 사용자가 장비와 소모품 데이터를 업데이트
3. 사용자가 `trips/2026-04-18-gapyeong.yaml` 을 준비
4. UI에서 해당 trip을 선택하고 분석 요청
5. 로컬 API가 전체 파일을 읽고 AI 백엔드 요청
6. 결과로 장비 추천, 유치원생 준비물, 바비큐/국물요리 추천, 주변 코스 추천 생성
7. 결과를 `outputs/2026-04-18-gapyeong-plan.md` 에 저장

### 예시 2. 장소만 먼저 정한 캠핑

1. 사용자가 장소만 있는 trip 파일 작성
2. UI에서 해당 trip을 선택
3. 로컬 API가 장소 기반 기본 추천 수행
4. 날짜 미입력으로 인한 제한을 결과에 표시
5. 이후 날짜가 정해지면 같은 trip 파일을 수정해 재분석

## 12. 향후 자동화 포인트

추후 아래를 자동화할 수 있다.

- trip 템플릿 생성 스크립트
- YAML 유효성 검사 스크립트
- 결과 Markdown 저장 스크립트
- 날씨 캐시 수집 스크립트
- 장소 캐시 수집 스크립트
- 특정 trip 재분석 명령

## 13. 현재 결정 사항

- 분석 실행 단위는 `trips/*.yaml` 파일이다
- 공통 사용자 데이터와 trip 요청 데이터를 분리한다
- 개인 준비물은 분석 결과로 생성한다
- 결과는 Markdown으로 저장한다
- 외부 정보는 선택적으로 결합한다
- v1은 로컬 웹 UI + 로컬 API 기반 반복 실행에 최적화한다
