# 캠핑 분석 워크플로우

## 1. 문서 목적

이 문서는 사용자가 UI에서 계획을 만들고, 장비를 점검하고, AI 보조와 분석을 거쳐, 최종적으로 히스토리로 아카이브하는 전체 흐름을 정의한다.

## 2. 핵심 원칙

- 분석의 기준 파일은 `trips/*.yaml`
- 장비와 링크는 계획과 별도 파일로 관리한다
- AI 보조는 제안만 하고 자동 저장하지 않는다
- 분석 결과는 `outputs/*.md` 로 저장한다
- 완료된 계획은 `history/*.yaml` 로 이동한다

## 3. 기본 실행 흐름

### 1단계. 기본 데이터 관리

사용자는 먼저 아래 데이터를 관리한다.

- `companions.yaml`
- `equipment/*.yaml`
- `preferences/*.yaml`
- `links.yaml`

현재 UI에서는 `캠핑 계획` 화면 안에서 동행자 프로필을 바로 추가/수정할 수 있다.

### 2단계. 캠핑 계획 작성

사용자는 `캠핑 계획` 화면에서 아래를 입력한다.

- 날짜
- 장소
- 동행자 ID
- 차량 정보
- 날씨와 취사/전기 조건
- 식단/이동 요청

입력한 동행자 ID가 아직 `companions.yaml` 에 없으면, 사용자는 같은 화면에서 즉시 등록할 수 있고 저장 시 기본 프로필로 자동 추가할 수도 있다.

### 3단계. AI 보조 사용

사용자는 저장된 계획을 기준으로 AI 보조에 질문한다.

AI 보조는:

- 지금 부족한 정보 경고
- 장비 보강 제안
- 점검 항목 제안
- 폼에서 먼저 바꿀 값 안내

를 반환한다.

### 4단계. 분석 실행

사용자는 `분석 실행` 을 누른다.

로컬 API는:

1. 계획 파일을 읽는다
2. 공통 데이터와 장비 파일을 읽는다
3. 검증 경고를 생성한다
4. 프롬프트와 문서를 결합한다
5. 로컬 `codex exec` 또는 fallback AI 백엔드를 호출한다
6. Markdown 결과를 UI에 반환한다

### 5단계. 결과 저장

분석 결과는 분석이 끝나면 자동으로 저장된다.

같은 `trip_id` 의 기존 결과가 있으면 같은 경로에 자동으로 덮어쓴다.

저장 위치:

- `.camping-data/outputs/<trip-id>-plan.md`

### 6단계. 히스토리 아카이브

캠핑이 끝난 뒤 사용자는 계획을 히스토리로 이동한다.

이때:

- `trips/<trip-id>.yaml` 는 계획 목록에서 제거된다
- `history/<trip-id>.yaml` 가 생성된다
- 결과 문서 경로는 히스토리에 참조로 남는다
