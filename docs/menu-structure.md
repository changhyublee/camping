# 메뉴 구조

## 1. 문서 목적

이 문서는 사용자가 반복적으로 쓰기 편한 `운영형 로컬 캠핑 관리자` 기준의 메뉴 구조를 정의한다.

목표:

- 분석 중심 단일 화면을 운영형 메뉴 구조로 재정리
- 장비, 계획, 히스토리, 외부 링크, 관리 설정의 책임을 분리
- 각 메뉴에서 어떤 CRUD와 AI 액션이 가능한지 고정

## 2. 최상위 메뉴

최상위 메뉴 6개는 유지하되, 실제 화면에서는 아래 3개 그룹으로 묶는다.

### 2.1 운영 허브

포함 메뉴:

- 대시보드
- 캠핑 계획
- 캠핑 히스토리

구성 원칙:

- 현재 상태 확인과 실제 실행 흐름을 한 묶음으로 둔다
- 사용자가 가장 자주 오가는 화면을 상단에 둔다
- 각 메뉴에는 이름만 아니라 `무엇을 하는지` 짧은 설명과 상태 메타를 함께 표시한다

### 2.2 준비 데이터

포함 메뉴:

- 장비 관리
- 외부 링크

구성 원칙:

- 계획을 실행하기 전에 준비해야 하는 데이터를 별도 묶음으로 분리한다
- 장비, 북마크처럼 비교적 안정적인 CRUD 화면을 계획 화면과 시각적으로 구분한다

### 2.3 관리 설정

포함 메뉴:

- 관리 설정

구성 원칙:

- 운영 기준값과 백업은 주 작업 화면과 떨어뜨려 둔다
- 카테고리, 백업 같은 시스템성 액션은 별도 섹션으로 인지되게 한다

### 2.4 대시보드

목적:

- 앱 첫 진입 허브
- 운영 현황을 빠르게 확인

핵심 요소:

- 예정된 캠핑 계획 수
- 최근 히스토리 수
- 점검 필요/재고 부족 항목 수
- 외부 링크 수
- 빠른 이동 버튼

### 2.5 장비 관리

목적:

- 보유 장비와 출발 전 점검 항목 관리

하위 섹션:

- 반복 장비
- 소모품
- 출발 전 점검

핵심 액션:

- 카테고리별 목록 보기
- 항목 추가
- 항목 수정
- 수량 변경
- 상태 변경
- 삭제
- 카테고리 셀렉트 사용

### 2.6 캠핑 계획

목적:

- 이번 캠핑 일정 작성과 분석 실행

핵심 요소:

- 계획 목록
- 새 계획 작성
- 날짜/장소/동행자/차량/조건 폼
- 동행자 프로필 인라인 관리
- AI 보조 대화
- 분석 실행
- 결과 자동 저장
- 히스토리 아카이브

원칙:

- 폼이 원본 입력의 기준이다
- AI는 제안을 생성하지만 자동 저장하지 않는다
- 사용자가 직접 제안을 적용해야 실제 파일이 바뀐다

### 2.7 캠핑 히스토리

목적:

- 완료된 계획의 결과를 기록으로 남기고 재활용

핵심 요소:

- 일정/지역/인원 요약
- 메모 수정
- 결과 문서 참조
- 삭제

원칙:

- 히스토리는 `계획 종료 후 아카이브` 로 생성한다
- 계획과 히스토리는 서로 다른 파일로 관리한다

### 2.8 외부 링크

목적:

- 날씨, 장소, 맛집, 장보기 등 참고 링크를 직접 관리

핵심 요소:

- 카테고리별 링크 목록
- 링크 추가/수정/삭제
- 새 탭 열기

### 2.9 관리 설정

목적:

- 장비 섹션별 관리 기준값과 운영 데이터 백업을 관리

핵심 요소:

- 수동 백업 실행
- 반복 장비/소모품/출발 전 점검 카테고리 목록
- 카테고리 표시 이름 수정
- 카테고리 추가/삭제
- 카테고리 코드 확인

## 3. 기본 사용자 흐름

### 흐름 A. 계획 만들기

1. 대시보드 또는 캠핑 계획에서 새 계획을 만든다
2. 날짜, 장소, 인원, 조건을 폼에 입력한다
3. AI 보조에 질문해 장비와 주의사항 제안을 받는다
4. 필요한 제안을 직접 반영한다
5. 분석을 실행해 결과를 확인한다

### 흐름 B. 장비 점검하기

1. 장비 관리로 이동한다
2. 반복 장비/소모품/점검 탭을 전환한다
3. 부족한 수량과 상태를 수정한다
4. 다시 계획 페이지로 돌아가 분석한다

### 흐름 C. 캠핑 종료 후 정리

1. 계획 페이지에서 히스토리 아카이브를 실행한다
2. 히스토리에서 실제 인원수와 메모를 보정한다
3. 필요 없는 히스토리는 삭제한다

## 4. 데이터 연결 기준

- `장비 관리` -> `equipment/*.yaml`
- `관리 설정` -> `equipment/categories.yaml`
- `캠핑 계획` -> `trips/*.yaml`
- `캠핑 히스토리` -> `history/*.yaml`
- `외부 링크` -> `links.yaml`
- `분석 결과` -> `outputs/*.md`

## 5. 구현 기준

- 첫 화면은 `대시보드`
- 네비게이션은 메뉴를 항상 드러내는 구조를 우선
- 메뉴는 `운영 허브`, `준비 데이터`, `관리 설정` 그룹으로 구분해 보여준다
- 각 메뉴 버튼에는 이름 외에 역할 설명과 상태 메타를 함께 둔다
- AI 보조는 `제안 생성 + 사용자 적용` 흐름을 유지
- 브라우저는 OpenAI API를 직접 호출하지 않음
