# 로컬 UI 전환 계획

## 1. 문서 목적

이 문서는 기존 `CLI 중심 분석 흐름` 을 `로컬 웹 UI + 로컬 API` 구조로 전환하기 위한 배경, 목표, 작업 목록을 정리한다.

## 2. 전환 배경

기존 문서는 Codex CLI를 직접 실행해 trip 파일을 분석하는 흐름을 중심으로 작성되어 있었다.

이 방식은 초기 검증에는 단순하지만 아래 문제가 있다.

- 비개발 사용자 기준 반복 사용성이 낮다
- 결과 비교와 재실행 흐름이 UI 없이 불편하다
- API 키 관리와 저장 흐름을 화면 경험과 분리하기 어렵다

따라서 v1 실행 구조를 아래처럼 바꾼다.

- 브라우저에서 실행하는 로컬 웹 UI
- AI 호출과 파일 저장을 담당하는 로컬 API
- 계속 유지되는 `.camping-data/` 기반 로컬 파일 저장

## 3. 유지할 것과 바뀌는 것

### 유지할 것

- 로컬 우선 구조
- `.camping-data/` 운영 데이터 저장
- 사람이 관리하는 YAML 입력
- Markdown 결과 저장
- trip 파일 중심 분석 단위
- 개인 준비물은 AI가 파생하는 추천 결과라는 도메인 규칙

### 바뀌는 것

- 주요 실행 주체를 `Codex CLI` 에서 `로컬 웹 UI + 로컬 API` 로 변경
- AI 호출 지점을 사용자 터미널 직접 실행에서 로컬 API 계층으로 이동
- 분석 결과 소비 방식을 터미널 출력 중심에서 화면 표시 + 파일 저장으로 변경

## 4. 목표 구조

```text
사용자
  -> 로컬 웹 UI
    -> 로컬 API
      -> docs/, prompts/, schemas/, .camping-data/ 읽기
      -> 로컬 codex CLI 또는 fallback AI 백엔드 호출
      -> outputs/*.md 저장
```

핵심 원칙:

- 브라우저는 AI 백엔드를 직접 호출하지 않는다
- 로컬 API가 기본적으로 `codex login` 세션을 통해 분석을 실행한다
- 필요 시에만 로컬 API가 `OPENAI_API_KEY` 를 사용한다
- UI는 입력과 상태 표시를 담당하고, 규칙 집행은 로컬 API가 담당한다

## 5. 단계별 작업 목록

### 1단계. 문서 기준 재정의

- `README.md` 실행 구조 변경
- `docs/technical-architecture.md` 갱신
- `docs/trip-analysis-workflow.md` 갱신
- `docs/mvp-scope.md` 갱신
- `docs/project-setup-checklist.md` 갱신
- `docs/directory-structure.md` 갱신
- `docs/prompt-design.md` 갱신

### 2단계. 저장소 구조 준비

- `apps/web/` 추가
- `apps/api/` 추가
- `shared/` 추가
- `.gitignore` 추가
- `.env.example` 추가

### 3단계. 로컬 API 준비

- trip 목록 조회 API 정의
- trip 상세 조회 API 정의
- 분석 실행 API 정의
- 결과 저장 API 정의
- YAML 로딩과 검증 계층 설계

### 4단계. UI 준비

- trip 목록/상세 화면
- 분석 실행 버튼과 상태 영역
- 경고와 오류 표시
- 결과 Markdown 렌더링
- 저장 액션

### 5단계. 첫 종단간 루프

- 예시 trip 파일 준비
- OpenAI 요청 1회 성공
- 결과 화면 노출
- 결과 Markdown 저장
- 재실행과 부분 수정 확인

## 6. v1 권장 구현 순서

1. 로컬 API에서 trip 조회와 분석 실행을 먼저 구현
2. UI에서 trip 선택과 결과 렌더링을 구현
3. prompts/ 와 schemas/ 를 연결
4. outputs 저장과 재실행 UX를 붙인다
5. 이후에만 CRUD UI 확장을 검토한다

## 7. 리스크와 대응

### 리스크 1. 원격 서버 설계로 과도하게 확장될 수 있음

대응:

- v1 범위를 로컬 웹 UI와 로컬 API로 제한한다
- 인증, 다중 사용자, DB는 제외한다

### 리스크 2. 브라우저에 API 키를 넣는 잘못된 구현이 생길 수 있음

대응:

- 로컬 API를 필수 계층으로 고정한다
- 환경변수 사용 규칙을 체크리스트와 API 문서에 포함한다

### 리스크 3. UI 도입 후 파일 기반 모델이 흐려질 수 있음

대응:

- UI가 `.camping-data/` 구조를 숨기지 않고 trip 중심으로 드러내게 한다
- 저장 경로와 파일 책임을 문서에 유지한다

## 8. 완료 기준

아래가 가능하면 전환 1단계가 완료된 것으로 본다.

1. 문서 기준이 모두 로컬 UI 구조와 일치한다
2. 저장소에 로컬 UI/API 구조를 둘 경로가 정해져 있다
3. API 키가 브라우저로 노출되지 않는 설계가 문서화되어 있다
4. UI에서 trip을 선택하고 분석 결과를 볼 수 있는 구현 계획이 고정되어 있다
