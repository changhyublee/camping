# UI 흐름

## 1. 문서 목적

이 문서는 로컬 웹 UI의 최소 화면 흐름과 사용자 액션을 정의한다.

목표:

- 구현 전에 화면 책임을 고정
- v1에서 필요한 최소 UX 범위를 정한다
- 전체 CRUD 관리자와 분석 실행 UI를 구분한다

## 2. v1 UI 원칙

- trip 분석 실행을 가장 먼저 지원한다
- 파일 기반 저장 모델을 숨기지 않는다
- 결과 문서를 읽기 쉽게 보여주는 것이 우선이다
- 장비/동행자/취향 전체 관리 화면은 후순위다

## 3. 주요 화면

### 화면 1. 홈 또는 trip 목록

목적:

- 분석 대상 trip 파일 선택

핵심 요소:

- trip 목록
- trip 제목, 날짜, 지역
- 새 trip 작성 안내 또는 템플릿 경로 안내
- 분석 시작 버튼

### 화면 2. trip 상세/분석 실행

목적:

- 선택한 trip의 핵심 정보를 확인
- 분석을 실행

핵심 요소:

- 기본 trip 정보
- 동행자 요약
- 조건 요약
- 경고 또는 누락 정보
- `분석 실행` 버튼
- `저장 후 실행` 옵션

### 화면 3. 분석 결과

목적:

- 결과 Markdown 표시
- 저장/재실행 제공

핵심 요소:

- 요약
- 추천 장비
- 연령대별 개인 준비물
- 부족한 물품
- 체크리스트
- 식단/요리
- 이동 중 추천
- 주변 추천
- 리스크와 한계
- 저장 버튼
- 재실행 버튼

## 4. 핵심 사용자 흐름

### 흐름 A. 기본 분석

1. 사용자가 홈에서 trip을 선택한다
2. UI가 `GET /api/trips/:tripId` 로 상세를 가져온다
3. 사용자가 `분석 실행` 을 누른다
4. UI가 `POST /api/analyze-trip` 을 호출한다
5. 결과를 Markdown 렌더링으로 보여준다

### 흐름 B. 저장 후 실행

1. 사용자가 `저장 후 실행` 옵션을 켠다
2. UI가 `save_output: true` 로 분석 요청한다
3. API가 저장까지 성공하면 결과 경로를 반환한다
4. UI가 저장 완료 상태를 보여준다
5. 저장만 실패하면 API는 Markdown 본문과 `OUTPUT_SAVE_FAILED` 를 함께 반환한다
6. UI는 결과 Markdown을 유지한 채 `warning` 상태로 저장 실패를 안내하고, 사용자가 다시 저장할 수 있게 한다

### 흐름 C. 결과 저장

1. 사용자가 저장하지 않고 먼저 결과를 확인한다
2. 이후 `저장` 버튼을 누른다
3. UI가 `POST /api/outputs` 를 호출한다
4. 저장 경로를 사용자에게 보여준다

### 흐름 D. 오류 처리

1. 사용자가 분석을 실행한다
2. API가 필수 입력 누락 또는 AI 백엔드 오류를 반환한다
3. UI가 오류 메시지와 수정 포인트를 보여준다
4. 사용자가 trip 또는 기본 데이터를 수정한 뒤 재실행한다

## 5. 상태 표시 규칙

권장 상태:

- `idle`
- `loading`
- `success`
- `warning`
- `error`

UI 원칙:

- `loading` 중에는 중복 실행을 막는다
- `warning` 은 분석은 성공했지만 정확도 제한이 있음을 뜻한다
- `warning` 은 결과 생성은 성공했지만 저장이 실패한 partial-success 상태에도 사용한다
- `error` 는 분석 본문 생성 자체가 완료되지 못했거나, 별도 저장 요청이 실패한 상태다

## 6. v1에서 제외하는 UI

- 장비 전체 CRUD 관리자
- 동행자 프로필 관리자
- 취향 설정 관리자
- 날씨/장소 캐시 관리자
- 로그인/권한 화면

## 7. 구현 메모

- 결과 영역은 Markdown 렌더러를 사용한다
- trip 목록은 `.camping-data/trips/` 파일 목록 기준으로 구성한다
- 경고 메시지는 결과 상단에 별도 영역으로 노출한다
- 저장 성공 시 출력 파일 경로를 함께 보여준다
