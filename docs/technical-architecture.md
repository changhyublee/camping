# 기술 아키텍처

## 1. 문서 목적

이 문서는 [`requirements.md`](/Users/leech/workspace/camping/camping/docs/requirements.md) 에 정리된 캠핑 추천 요구사항을 실제로 구현하기 위한
초기 기술 방향을 정리한다.

현재 목표는 별도 서버를 개발하지 않고도,
사용자 데이터와 캠핑 조건을 바탕으로 Codex CLI가 분석하여
캠핑 준비물, 개인 준비물, 요리, 이동 동선, 주변 즐길거리까지 추천할 수 있는 구조를 잡는 것이다.

## 2. 핵심 전제

- 초기 버전은 1인 또는 소규모 개인 사용을 우선한다.
- 웹 서버, DB 서버, 인증 서버 없이 동작하는 구조를 우선 검토한다.
- 사용자는 Codex CLI를 통해 로컬 파일을 읽고 분석 결과를 받는다.
- 사용자 데이터는 사람이 직접 수정하거나, 추후 로컬 CLI 도구로 관리할 수 있어야 한다.
- 최신 날씨, 장소, 맛집 정보처럼 변동성이 큰 데이터는 추후 선택적으로 연동할 수 있다.

## 3. 아키텍처 선택지 검토

### 선택지 A. GitHub 저장소를 운영 저장소로 사용

구성:

- 사용자 입력값을 GitHub repo 파일에 저장
- Codex CLI가 원격 저장소 기준으로 읽고 분석
- 변경이 생길 때마다 commit/push 필요

장점:

- 백업과 변경 이력 관리가 쉽다
- 여러 기기에서 같은 데이터를 볼 수 있다
- 협업 구조로 확장하기 쉽다

단점:

- 매번 네트워크가 필요하다
- push/pull 과정이 번거롭다
- 개인 데이터가 repo에 쌓여 운영 저장소와 백업 저장소의 역할이 섞인다
- 토큰 사용량과 네트워크 비용이 증가한다
- 빠른 반복 입력과 분석에는 비효율적이다

판단:

- `백업/공유` 용도로는 유용하지만, `실시간 운영 저장소` 로는 적합하지 않다

### 선택지 B. 로컬 파일을 운영 저장소로 사용

구성:

- 사용자 데이터는 로컬 디렉토리에 파일 형태로 저장
- Codex CLI가 로컬 파일을 직접 읽고 분석
- 결과도 로컬 파일로 저장 가능

장점:

- 네트워크 없이 동작한다
- 가장 단순하고 빠르다
- 토큰과 네트워크 비용을 줄일 수 있다
- 사용자 데이터와 코드 저장소의 책임을 분리하기 쉽다
- 초기 버전에 서버가 전혀 필요 없다

단점:

- 기기 간 자동 동기화가 없다
- 백업 전략을 별도로 정해야 한다
- 여러 사용자가 동시에 쓰는 구조에는 적합하지 않다

판단:

- 현재 프로젝트의 초기 목표에 가장 적합하다

### 선택지 C. 하이브리드 구조

구성:

- 운영 데이터는 로컬 파일을 기준으로 관리
- 필요할 때만 GitHub에 수동 백업 또는 export

장점:

- 평소에는 빠르고 단순하다
- 필요 시 백업과 이력 관리가 가능하다
- 운영과 백업의 역할을 분리할 수 있다

단점:

- 로컬과 백업 데이터의 동기화 규칙이 필요하다
- export/import 규칙을 문서화해야 한다

판단:

- 현재 시점의 최적안이다

## 4. 권장안

초기 버전은 `로컬 우선(Local-first) + 선택적 GitHub 백업` 구조를 권장한다.

핵심 원칙:

- GitHub는 운영 저장소가 아니라 `문서/코드 저장소` 로 사용한다
- 사용자 데이터는 로컬 파일로 관리한다
- Codex CLI는 로컬 파일을 읽어 분석한다
- 필요할 때만 사용자가 백업용으로 GitHub에 export 한다

이 방식이 적합한 이유:

- 이 프로젝트는 고트래픽 서비스가 아니라 `개인 의사결정 지원 도구` 에 가깝다
- 핵심은 빠르게 조건을 바꾸고 분석을 반복하는 것이다
- 서버, DB, 인증 없이도 핵심 가치 검증이 가능하다
- GitHub를 운영 DB처럼 쓰는 것은 비용과 절차가 과하다

## 5. 로컬 저장 경로 제안

### 권장안: 프로젝트 내부 숨김 디렉토리

권장 경로:

```text
./.camping-data/
```

장점:

- 프로젝트와 함께 관리하기 쉽다
- Codex CLI가 같은 작업 디렉토리에서 읽기 쉽다
- 문서, 스키마, 스크립트와 연결하기 쉽다
- 나중에 `.gitignore` 로 쉽게 제외할 수 있다

주의:

- 반드시 Git 추적 대상에서 제외해야 한다

### 대안: 사용자 홈 디렉토리

대안 경로:

```text
~/.camping/
```

장점:

- 여러 저장소나 여러 프로젝트에서 공통 사용자 데이터를 재사용하기 쉽다
- 코드 저장소와 개인 데이터를 더 명확히 분리할 수 있다

단점:

- 프로젝트 문서와 실제 데이터가 물리적으로 떨어진다
- 초기 구현과 디버깅이 조금 더 번거롭다
- Codex CLI 작업 흐름상 프로젝트 내부 경로보다 불편할 수 있다

결론:

- `v1 기본값은 ./.camping-data/`
- 향후 필요하면 `~/.camping/` 지원을 추가

## 7. 데이터 저장 방식 제안

### 사용자 입력 파일

권장 포맷:

- `YAML`

이유:

- 사람이 직접 읽고 수정하기 쉽다
- 구조화된 데이터를 표현하기 충분하다
- JSON보다 주석과 가독성이 좋다

예외:

- 기계가 생성하는 캐시 데이터는 `JSON` 으로 저장해도 된다

예시 분리:

- 사용자가 직접 관리하는 데이터: `YAML`
- 외부 조회 결과 캐시: `JSON`
- 분석 결과 문서: `Markdown`

## 8. Codex CLI 중심 동작 흐름

### 기본 흐름

1. 사용자가 로컬 데이터 파일을 입력 또는 수정한다
2. 사용자가 캠핑 요청 조건을 입력한다
3. 스크립트가 `README`, `docs`, `prompts`, `.camping-data` 를 모아 분석용 컨텍스트를 구성한다
4. Codex CLI가 컨텍스트를 읽고 캠핑 추천 결과를 생성한다
5. 결과를 터미널에 출력하고 필요하면 `outputs/` 에 저장한다

### 분석 시 사용하는 입력 요소

- 캠핑 날짜
- 캠핑 장소
- 동행 인원 및 연령대
- 보유 캠핑 장비
- 소모품 잔량
- 장비 사전 점검 상태
- 음식 선호 및 요리 계획
- 여행/맛집/즐길거리 선호
- 필요 시 수동 입력한 날씨 정보
- 추후 선택적으로 수집한 외부 날씨/장소 정보

## 9. 서버가 없어도 되는 이유

현재 요구사항은 아래 특성을 가진다.

- 개인 맞춤 추천이 핵심이다
- 대량 동시 접속 처리가 필요하지 않다
- 강한 트랜잭션 정합성이 필요하지 않다
- 파일 기반 저장으로도 충분히 관리 가능하다
- Codex CLI가 이미 분석 실행 환경 역할을 할 수 있다

즉, 지금 단계에서 서버를 먼저 만들면
핵심 가치 검증보다 인프라 작업이 앞서게 된다.

## 10. 추후 서버가 필요해지는 시점

아래 조건이 생기면 서버 도입을 재검토한다.

- 여러 사용자가 동시에 사용하는 경우
- 모바일 앱/웹 앱에서 같은 데이터를 실시간 공유해야 하는 경우
- 최신 날씨, 장소, 맛집 정보를 자동 수집하고 캐시해야 하는 경우
- 사용자 계정, 권한, 동기화, 백업 정책이 필요해지는 경우

그 전까지는 파일 기반 구조로 충분하다.

## 12. 현재 결정 사항

- 서버 없는 구조로 시작한다
- GitHub는 운영 저장소가 아니라 코드/문서 저장소로 사용한다
- 사용자 데이터는 로컬 파일로 저장한다
- 기본 로컬 경로는 `./.camping-data/` 로 한다
- 로컬 데이터는 사람이 편하게 다룰 수 있도록 `YAML` 을 우선 검토한다
- 분석 실행은 Codex CLI 중심으로 설계한다
