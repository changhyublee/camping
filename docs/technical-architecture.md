# 기술 아키텍처

## 1. 문서 목적

이 문서는 [`requirements.md`](requirements.md) 에 정리된 캠핑 추천 요구사항을 실제로 구현하기 위한 기술 방향을 정리한다.

현재 목표는 별도 원격 서버를 개발하지 않고도, 로컬 웹 UI와 로컬 API를 통해 사용자 데이터와 캠핑 조건을 입력하고 기본적으로 로컬 `codex CLI` 를 사용해 캠핑 준비물, 개인 준비물, 요리, 이동 동선, 주변 즐길거리까지 추천할 수 있는 구조를 잡는 것이다. 필요하면 OpenAI Responses API를 fallback 백엔드로 사용할 수 있다.

## 2. 핵심 전제

- 초기 버전은 1인 또는 소규모 개인 사용을 우선한다.
- 원격 웹 서버, DB 서버, 인증 서버 없이 동작하는 구조를 우선 검토한다.
- 사용자는 브라우저 UI를 통해 로컬 파일 기반 데이터를 보고 분석 결과를 받는다.
- 로컬 API는 파일 시스템 접근과 AI 백엔드 호출을 담당한다.
- 사용자 데이터는 사람이 직접 수정하거나, 추후 로컬 UI로 관리할 수 있어야 한다.
- 최신 날씨, 장소, 맛집 정보처럼 변동성이 큰 데이터는 추후 선택적으로 연동할 수 있다.

## 3. 아키텍처 선택지 검토

### 선택지 A. 브라우저 UI가 OpenAI API를 직접 호출

구성:

- 브라우저가 OpenAI API를 직접 호출
- 로컬 파일은 별도 브리지 없이 간접 처리
- API 키를 클라이언트에 보관해야 함

장점:

- 구현 단계가 짧아 보인다
- 초기 데모는 빠르게 만들 수 있다

단점:

- API 키 노출 위험이 크다
- 브라우저에서 로컬 파일 시스템과 저장 규칙을 일관되게 다루기 어렵다
- 프롬프트, 스키마, 저장 정책을 서버 측에서 통제하기 어렵다

판단:

- 보안과 구조 안정성 측면에서 v1 권장안으로 부적합하다

### 선택지 B. 로컬 웹 UI + 로컬 API + 로컬 파일 저장

구성:

- 브라우저 UI가 로컬 API에 요청
- 로컬 API가 `.camping-data/` 와 문서/프롬프트를 읽어 로컬 `codex exec` 호출
- 결과를 UI에 반환하고 필요하면 Markdown으로 저장

장점:

- 브라우저에 API 키를 둘 필요가 없고, 기본적으로 로컬 `codex login` 세션을 재사용할 수 있다
- 로컬 파일 저장과 결과 저장 규칙을 통제하기 쉽다
- 브라우저 UX와 파일 기반 구조를 함께 유지할 수 있다
- 추후 검증, 캐시, 재시도, 저장 정책을 API 계층에 추가하기 쉽다

단점:

- 로컬 프로세스 두 개 이상을 띄워야 할 수 있다
- CLI 단일 흐름보다 초기 구성 요소가 늘어난다

판단:

- 현재 프로젝트 목표에 가장 적합하다

### 선택지 C. 데스크톱 앱 셸까지 포함한 패키지 앱

구성:

- Electron 또는 Tauri 같은 데스크톱 런타임 사용
- UI와 로컬 API를 하나의 앱 패키지로 배포

장점:

- 로컬 파일 권한과 설치 경험을 더 통합하기 쉽다
- 향후 배포 경험을 개선하기 좋다

단점:

- v1 시점에는 빌드와 배포 복잡도가 커진다
- 현재 저장소에는 아직 패키지 앱이 필요할 정도의 요구가 없다

판단:

- 추후 확장 후보로 두되 v1 기본값으로 채택하지 않는다

## 4. 권장안

초기 버전은 `로컬 우선(Local-first) + 로컬 웹 UI + 로컬 API + 선택적 GitHub 백업` 구조를 권장한다.

핵심 원칙:

- GitHub는 운영 저장소가 아니라 `문서/코드 저장소` 로 사용한다
- 사용자 데이터는 로컬 파일로 관리한다
- UI는 브라우저에서 실행하되 실제 AI 백엔드 호출은 로컬 API가 수행한다
- 로컬 API는 기본적으로 `codex exec` 를 통해 분석 결과를 생성한다
- 필요하면 fallback 으로 `Responses API` 를 사용할 수 있다
- 필요할 때만 사용자가 백업용으로 GitHub에 export 한다

## 5. 상위 구조

권장 구조:

```text
브라우저 UI
  -> 로컬 API
    -> prompts/, docs/, schemas/ 참조
    -> .camping-data/ YAML 로드
  -> 로컬 codex CLI 호출
    -> Markdown 결과 저장
```

역할 분리:

- `apps/web`
  - trip 선택, 분석 실행, 결과 표시, 저장 요청
- `apps/api`
  - 입력 검증, 컨텍스트 조합, AI 백엔드 호출, 파일 저장
- `.camping-data`
  - 운영 데이터 저장
- `prompts`
  - OpenAI 분석 프롬프트
- `schemas`
  - YAML/입력 검증 규칙

## 6. 로컬 저장 경로

### 권장안: 프로젝트 내부 숨김 디렉토리

권장 경로:

```text
./.camping-data/
```

장점:

- 프로젝트와 함께 관리하기 쉽다
- 로컬 API가 같은 작업 디렉토리에서 읽기 쉽다
- 문서, 프롬프트, 스키마, UI와 연결하기 쉽다
- `.gitignore` 로 쉽게 제외할 수 있다

주의:

- 반드시 Git 추적 대상에서 제외해야 한다

### 대안: 사용자 홈 디렉토리

대안 경로:

```text
~/.camping/
```

장점:

- 여러 저장소나 여러 실행 환경에서 공통 사용자 데이터를 재사용하기 쉽다
- 코드 저장소와 개인 데이터를 더 명확히 분리할 수 있다

단점:

- 문서와 실제 데이터가 물리적으로 떨어진다
- 초기 구현과 디버깅이 조금 더 번거롭다

결론:

- `v1 기본값은 ./.camping-data/`
- 향후 필요하면 `~/.camping/` 지원을 추가

## 8. 데이터 저장 방식

사용자 입력 파일:

- `YAML`

분석 결과:

- `Markdown`

외부 조회 또는 캐시:

- `JSON`

원칙:

- 사람이 관리하는 데이터는 YAML
- 로컬 API가 생성하는 결과는 Markdown
- 시점 의존 캐시는 JSON

## 9. OpenAI 연동 원칙

- 브라우저에서 직접 OpenAI API를 호출하지 않는다
- 기본 AI 백엔드는 로컬 `codex login` 세션을 사용하는 `codex exec` 다
- 필요하면 로컬 API가 `OPENAI_API_KEY` 환경변수를 사용해 fallback 으로 호출할 수 있다
- 모델명은 코드 또는 환경변수에서 설정 가능하게 하되, 문서에서는 특정 모델명보다 API 계약과 입력 구조를 우선한다
- 프롬프트는 `prompts/` 디렉토리에서 로드하고, 요청별로 trip 데이터와 결합한다

## 10. UI 중심 동작 흐름

1. 사용자가 로컬 UI에서 trip을 선택하거나 입력한다
2. 로컬 API가 관련 YAML 파일을 읽고 검증한다
3. 로컬 API가 문서, 프롬프트, trip 데이터, 선택적 캐시를 조합한다
4. 로컬 codex CLI 또는 fallback AI 백엔드가 캠핑 추천 결과를 생성한다
5. 로컬 API가 결과를 UI에 반환하고 필요하면 `outputs/` 에 저장한다
6. 사용자는 결과를 검토하고 입력을 수정해 재실행한다

## 11. 원격 서버가 없어도 되는 이유

현재 요구사항은 아래 특성을 가진다.

- 개인 맞춤 추천이 핵심이다
- 대량 동시 접속 처리가 필요하지 않다
- 강한 트랜잭션 정합성이 필요하지 않다
- 파일 기반 저장으로도 충분히 관리 가능하다
- 로컬 API 하나로 파일 저장과 AI 백엔드 호출을 함께 처리할 수 있다

즉, 지금 단계에서 원격 서버를 먼저 만들면 핵심 가치 검증보다 인프라 작업이 앞서게 된다.

## 12. 추후 원격 서버가 필요해지는 시점

아래 조건이 생기면 원격 서버 도입을 재검토한다.

- 여러 사용자가 동시에 사용하는 경우
- 모바일 앱/웹 앱에서 같은 데이터를 실시간 공유해야 하는 경우
- 사용자 계정, 권한, 동기화, 백업 정책이 필요해지는 경우
- 외부 데이터 수집과 캐시 갱신을 중앙에서 자동 운영해야 하는 경우

그 전까지는 로컬 웹 UI + 로컬 API 구조로 충분하다.

## 13. 초기 구현 권장 범위

v1에서는 아래만 우선 구현한다.

- 로컬 UI에서 trip 목록과 trip 상세를 볼 수 있는 구조
- trip 기반 분석 실행 버튼과 상태 표시
- 로컬 API 입력 검증
- 로컬 codex CLI 호출
- 결과 Markdown 렌더링
- 결과 Markdown 저장

v1에서 제외해도 되는 항목:

- 원격 API 서버
- DB 서버
- 로그인/인증
- 다중 사용자 협업
- 모바일 앱 패키징
- 외부 API 자동 연동 필수화

## 14. 현재 결정 사항

- 원격 서버 없는 구조로 시작한다
- GitHub는 운영 저장소가 아니라 코드/문서 저장소로 사용한다
- 사용자 데이터는 로컬 파일로 저장한다
- 기본 로컬 경로는 `./.camping-data/` 로 한다
- 기본 실행 구조는 `로컬 웹 UI + 로컬 API + 로컬 codex CLI` 로 한다
- 브라우저는 AI 백엔드를 직접 호출하지 않는다
- 분석 결과는 Markdown 문서로 저장한다
